# X68000 スプライト制御

X68000のスプライト機能は、CPUに描画負荷をかけることなく独立したプレーンとして複数のオブジェクトを合成・高速表示するハードウェア機能。最大128個のスプライトを動的に表示可能。

## 1. ハードウェア配置とメモリマップ

スプライト制御用のRAMとPCG（プログラマブル・キャラクタ・ジェネレータ）エリアは以下の通り。

| 領域 | 開始アドレス | サイズ | 役割 |
| --- | --- | --- | --- |
| **スプライト属性テーブル (SAT)** | `$EB0000` | 1KB | 128個のスプライトの座標、パターンのインデックス、表示優先順位（8バイト/オブジェクト） |
| **PCG RAM** | `$EB8000` | 32KB | スプライト表示用パターンデータ（16×16ドット単位） |

### 座標系と球面スクロール

スプライト画面全体を移動させるための「スプライトスクロールレジスタ」は `$E80000` 系に存在する。
スプライト座標と画面スクロールの同期は、以下の論理式で行う。

```text
Display_X = (Sprite_SAT_X - Global_Scroll_X) & 511
```

X68000の画面空間は上下左右がつながった球面スクロール（ラップアラウンド）の特性を持つため、画面外へ消えたスプライトが反対側から出現する現象を制御するためには、このマスク計算を考慮して座標値を管理する。

## 2. スプライト属性テーブル (SAT) のC言語抽象化

SAT上のデータ構造は8バイト（4ワード）単位で、以下の構造体を用いて制御する。

### 書籍サンプル版 (SP_CTRL ビットフィールド共用体)

「Game Programming with GCC」の sp_test.c で使用されている構造体。
GCC m68k のビットフィールドは LSB 側から割り当てられるため、ハードウェアレジスタのビット配置と一致する。

```c
typedef struct {
    short int sp_x;
    short int sp_y;
    union {
        short int dummy;
        struct {
            unsigned v_invert : 1;   /* bit 0: 上下反転 */
            unsigned h_invert : 1;   /* bit 1: 左右反転 */
            unsigned dummy    : 2;   /* bit 2-3: 予約 */
            unsigned color    : 4;   /* bit 4-7: カラーパレット番号 */
            unsigned sp_code  : 8;   /* bit 8-15: パターン番号 */
        } sp;
    } sp_ctrl;
    union {
        short int dummy;
        struct {
            unsigned dummy   : 13;
            unsigned ext     :  1;
            unsigned pwr     :  2;   /* プライオリティ (0-3) */
        } sp;
    } sp_pwr;
} SP_CTRL;

/* 使用例 */
SP_CTRL sp_disp[128];
sp_disp[0].sp_x = 100;
sp_disp[0].sp_y = 50;
sp_disp[0].sp_ctrl.sp.sp_code = 0;  /* パターン0 */
sp_disp[0].sp_ctrl.sp.color = 1;    /* パレット1 */
sp_disp[0].sp_pwr.sp.pwr = 3;      /* 最前面 */
```

### シンプル版 (SPRITE_ATTR)

```c
/* スプライト属性構造体 (アライメントとビット幅をハードウェアに準拠) */
typedef struct {
    unsigned short x;     /* X座標 (下位10ビット有効) */
    unsigned short y;     /* Y座標 (下位10ビット有効) */
    unsigned short code;  /* パターン番号 (PCGのインデックス) */
    unsigned short attr;  /* 属性フラグ (左右反転、上下反転、カラーパレット番号、プライオリティ) */
} SPRITE_ATTR;

/* 128個のスプライト属性テーブルへの直接ポインタ */
volatile SPRITE_ATTR *const SAT_ADDR = (volatile SPRITE_ATTR *)0xeb0000;
```

> **[重要]** elf2x68k 環境では、スプライト IOCS 呼び出し (SP_INIT, SP_CGCLR, SP_ON, SP_PALET) はライブラリ関数 (`_iocs_sp_*`) が動作しないため、全て raw `trap #15` で呼び出すこと。詳細は `19_elf2x68kクロス開発.md` を参照。

> [!IMPORTANT]
> **volatile の重要性**
> レジスタポインタには必ず `volatile` キーワードを付与すること。GCCの最適化（`-O2` や `-O3`）が適用された際、ループ内などでの書き込みが最適化で省略されてしまい、ハードウェアレジスタへの指示が消失するバグを防ぐため。
> **奇数番地アクセスの禁止 (Address Error)**
> 68000 CPUは奇数番地（奇数アドレス）へのワード/ロングワードアクセスを許可しない。SATへの書き込みは常に偶数アドレスから始まるように、アライメントとキャスト (`volatile SPRITE_ATTR *`) を適切に管理すること。

## 3. PCG（パターンデータ）の書き込みとビッグエンディアン

PCG RAM（`$EB8000`〜）は、16×16ドット単位のビットマップデータを保持する。

- 256色モード時は 1ドットあたり 1バイト（計256バイト）。
- 68000は**ビッグエンディアン**であるため、PCや他のリトルエンディアン環境で作成したバイナリデータをそのままロードしてワード転送を行う場合は、バイトスワップが必要になる。
- 実際のPCG書き込み処理は、スプライトレジスタの制御（レジスタへのアクセス許可）を行ってから実施する。

### PCG書き込みのコード例（[sp_test.c](file:///Users/scott/repo/sokoide/agents/skills/x68k-master/resources/samples/sp_test.c) より抜粋）

```c
typedef union {
    unsigned short dummy;
    struct {
      unsigned pos0: 4;
      unsigned pos1: 4;
      unsigned pos2: 4;
      unsigned pos3: 4;
    } sp;
} SP_REG_U;

static void def_sp(int pat_no, unsigned char *sp_dat) {
   SP_REG_U *sp_reg0 = (SP_REG_U *) 0xeb8000 + (128/sizeof (SP_REG_U)) * pat_no;
   SP_REG_U *sp_reg1 = (SP_REG_U *) 0xeb8040 + (128/sizeof (SP_REG_U)) * pat_no;
   int i;
   
   /* スプライト制御レジスタを開き、CPUアクセスを許可する */
   *(short *)0xeb0808 &= 0xfdfc;
   
   for (i = 0; i < 16; i++) {
        SP_REG_U tem;
        tem.sp.pos0 = *sp_dat++;
        tem.sp.pos1 = *sp_dat++;
        tem.sp.pos2 = *sp_dat++;
        tem.sp.pos3 = *sp_dat++;
        *sp_reg0++ = tem;
        tem.sp.pos0 = *sp_dat++;
        tem.sp.pos1 = *sp_dat++;
        tem.sp.pos2 = *sp_dat++;
        tem.sp.pos3 = *sp_dat++;
        *sp_reg0++ = tem;

        tem.sp.pos0 = *sp_dat++;
        tem.sp.pos1 = *sp_dat++;
        tem.sp.pos2 = *sp_dat++;
        tem.sp.pos3 = *sp_dat++;
        *sp_reg1++ = tem;
        tem.sp.pos0 = *sp_dat++;
        tem.sp.pos1 = *sp_dat++;
        tem.sp.pos2 = *sp_dat++;
        tem.sp.pos3 = *sp_dat++;
        *sp_reg1++ = tem;
   }
   /* スプライト表示を有効化 */
   *(short *)0xeb0808 |= 0x0200;
}
```

## 4. 垂直帰線期間（V-Sync）同期とダブルバッファリング

表示中に直接SAT（`$EB0000`）を書き換えると、走査線が通過するタイミングと重なり、画面のティアリングやフリッカー（表示崩れ）を引き起こす。
これを回避するため、**「仮想SATによるダブルバッファリング」**と**「V-Sync同期」**を組み合わせる。

- **仮想SATダブルバッファリング**:
  メインメモリ（RAM）上にスプライト属性テーブルのバッファ（仮想SAT）を用意し、フレームロジック（座標更新等）はすべて仮想SATに対して行う。
- **V-Sync同期書き込み**:
  非表示期間（V-Blank期間）の発生タイミングに合わせて、仮想SATから本物のSATへと `memcpy` または高速ループで一括転送する。

### V-Sync同期の3つの実装手法

#### 手法A: VDISPST 割り込みを利用する（最も標準的でクリーン）

Human68kのIOCSコール `VDISPST` を用いて、垂直帰線割り込みベクタに自作の描画・同期ハンドラを登録する。

- 登録処理例: `VDISPST(vsync_disp_handler, 0, 1)`
- 解除処理例: `VDISPST(0, 0, 0)`（終了時に必ず元に戻すこと）
- 実装コードは [sp_test.c](file:///Users/scott/repo/sokoide/agents/skills/x68k-master/resources/samples/sp_test.c) を参照。

#### 手法B: ONTIME タイマーを利用する（ポーリングウェイト）

IOCSコール `ONTIME()` （ミリ秒カウンタ）をポーリングで監視し、一定時間経過（フレーム周期）を待つ。

- 利点: 割り込みハンドラを定義する必要がなく実装が簡単。
- 欠点: 走査線の厳密な非表示期間との同期は難しく、描画のチラつきが残る可能性がある。
- 実装コードは [sp_test0.c](file:///Users/scott/repo/sokoide/agents/skills/x68k-master/resources/samples/sp_test0.c) を参照。

#### 手法C: MFP GPIP4 ($E88001 bit4) ポーリング

I/OポートのMFPレジスタ（`$E88001`）を監視して、VBLANK信号の立ち上がり・立ち下がりを直接ループで待つ。

```c
char volatile *mpf = (char *)0xe88001;
while (!((*mpf) & 0x10)); /* 立ち上がり待ち */
while ((*mpf) & 0x10);    /* 立ち下がり待ち */
```

- 利点: 割り込みを使わずに、ハードウェア走査線との厳密なV-Sync同期を低オーバーヘッドで実行できる。
- 欠点: ポーリング待ちの間CPUを100%占有するため、省電力やマルチタスク環境（他の割り込みハンドリングなど）とトレードオフになる。
- 実装コードは [sp_test1.c](file:///Users/scott/repo/sokoide/agents/skills/x68k-master/resources/samples/sp_test1.c) を参照。
