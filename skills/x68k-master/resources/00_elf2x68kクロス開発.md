# elf2x68k クロス開発環境

elf2x68k (macOS/Linux → X68000 Human68k クロースコンパイラ) での実践的知見。
PX68k エミュレータでの検証結果を含む。

## 1. ツールチェーン構成

- コンパイラ: `m68k-xelf-gcc` (GCC, m68k-elf ターゲット)
- リンカ: `m68k-xelf-gcc` + elf2x68k コンバータ
- ライブラリ: `libx68kiocs.a`, `libx68kdos.a` (IOCS/DOS スタブ)
- 共通フラグ: `-O2 -m68000 -Wall`

## 2. IOCS ライブラリ関数について

elf2x68k 付属の `libx68kiocs.a` に含まれる IOCS ライブラリ関数は、逆アセンブル検証および**実機テストにて正常動作を確認済み**。

```asm
_iocs_sp_init:  moveq #-64,%d0;  trap #15; rts   ← D0=0xC0
_iocs_sp_on:    moveq #-63,%d0;  trap #15; rts   ← D0=0xC1
_iocs_sp_cgclr: movel %sp@(4),%d1; moveq #-61,%d0; trap #15; rts  ← D0=0xC3
_iocs_crtmod:   movel %sp@(4),%d1; moveq #16,%d0;  trap #15; rts  ← D0=0x10
_iocs_gpalet:   moveml %sp@(8),%d1-%d2; moveq #-108,%d0; trap #15; rts  ← D0=0x94
_iocs_g_clr_on: moveq #-112,%d0; trap #15; rts   ← D0=0x90
_iocs_spalet:   moveml %sp@(12),%d1-%d3; moveq #-49,%d0; trap #15; rts  ← D0=0xCF
```

**注意**: `_iocs_spalet` (SP_PALET) はライブラリに含まれるがヘッダに宣言がない。自前で宣言が必要:

```c
extern int _iocs_spalet(int pal_code, int block, int col_code);
```

### raw trap #15 インラインアセンブラ（代替手法）

ライブラリ関数が使える場面でも、raw trap を使うことで IOCS 呼び出しをインライン化（関数呼び出しオーバーヘッドなし）できる。

```c
/* 引数なし IOCS 呼び出し */
{
    register int trapNo asm("d0") = 0xC0; /* SP_INIT */
    asm("trap #15" :: "d"(trapNo));
}

/* 引数1個 IOCS 呼び出し */
{
    register int trapNo asm("d0") = 0x10; /* CRTMOD */
    register int mode   asm("d1") = 10;
    asm("trap #15" :: "d"(trapNo), "d"(mode));
}

/* 引数2個 IOCS 呼び出し */
{
    register int trapNo asm("d0") = 0x94; /* GPALET */
    register int pal    asm("d1") = 1;
    register int col    asm("d2") = 0xFBE1; /* white */
    asm("trap #15" :: "d"(trapNo), "d"(pal), "d"(col));
}

/* 引数4個 IOCS 呼び出し (SP_PALET) */
{
    register int trapNo   asm("d0") = 0xCF;
    register int pal_code asm("d1") = 0x80000000 | 0; /* bit31=set */
    register int block    asm("d2") = 1;
    register int col_code asm("d3") = 21140;
    asm("trap #15" :: "d"(trapNo), "d"(pal_code), "d"(block), "d"(col_code));
}
```

> **[注意]** `"+r"` 出力制約や clobber リストを追加すると `asm specifier conflicts with asm clobber list` エラーになる。上記の入力のみ制約 `"d"()` のパターンを使用すること。

## 3. PCG パッキング: GCC m68k ビットフィールドの LSB 割り当て（重要）

スプライトが表示されない問題の真の原因は IOCS 呼び出しではなく、**PCG（パターンデータ）のパッキング方式**にあった。

### GCC m68k のビットフィールドは LSB から割り当て

68000 はワード内のバイト順序がビッグエンディアンだが、**ビットフィールドの割り当て方向はコンパイラ依存**であり、GCC m68k は **LSB ファースト**である:

```c
typedef union {
    unsigned short dummy;
    struct {
        unsigned pos0: 4;  /* bits 0-3  (LSB) */
        unsigned pos1: 4;  /* bits 4-7 */
        unsigned pos2: 4;  /* bits 8-11 */
        unsigned pos3: 4;  /* bits 12-15 (MSB) */
    } sp;
} SP_REG_U;
```

### 手動パッキングで陥りやすい罠

68000 がビッグエンディアンであるため、MSB 側に先頭ピクセルを配置するよう手動シフト演算を書いてしまう:

```c
/* 間違い: MSB側に sp_dat[0] を置く（ビッグエンディアン直感） */
uint16_t tem = (sp_dat[0] << 12) | (sp_dat[1] << 8) | (sp_dat[2] << 4) | sp_dat[3];
```

しかし X68000 スプライトコントローラは **LSB 側に先頭ピクセルを期待する**。正しい手動パッキングは:

```c
/* 正しい: LSB側に sp_dat[0] を置く（bitfieldと同じ） */
uint16_t tem = (sp_dat[3] << 12) | (sp_dat[2] << 8) | (sp_dat[1] << 4) | sp_dat[0];
```

### 推奨: SP_REG_U ビットフィールド共用体を使う

書籍「Game Programming with GCC」の `SP_REG_U` 共用体を使えば、ビット配置を意識せずに正しくパッキングできる。**書籍の構造体にバグはない。そのまま使用してよい。**

```c
SP_REG_U tem;
tem.sp.pos0 = *sp_dat++;  /* 自動的に bits 0-3 に配置 */
tem.sp.pos1 = *sp_dat++;  /* 自動的に bits 4-7 に配置 */
tem.sp.pos2 = *sp_dat++;  /* 自動的に bits 8-11 に配置 */
tem.sp.pos3 = *sp_dat++;  /* 自動的に bits 12-15 に配置 */
*sp_reg0++ = tem;
```

### SP_CTRL も同じ LSB ファースト原則

SP_CTRL 構造体の sp_ctrl も LSB ファースト:

```c
struct {
    unsigned v_invert : 1;  /* bit 0 */
    unsigned h_invert : 1;  /* bit 1 */
    unsigned dummy    : 2;  /* bits 2-3 */
    unsigned color    : 4;  /* bits 4-7 */
    unsigned sp_code  : 8;  /* bits 8-15 */
} sp;
```

手動パッキング `(code << 8) | (color << 4)` はこの配置と一致するため正しい。ただし SP_REG_U と違い、こちらは直感通りの順序になるので手動でも間違いにくい。

## 4. IOCS 関数番号一覧

`libx68kiocs.a` を逆アセンブルして確認した実際の IOCS 関数番号 (D0 レジスタ値)。

### 画面制御系

| 関数名 | IOCS番号 (D0) | 引数 | 備考 |
| --- | --- | --- | --- |
| CRTMOD | 0x10 | D1=モード番号 | 画面モード設定 |
| B_CUROFF | 0x1F | なし | テキストカーソルOFF |
| B_CURON | 0x1E | なし | テキストカーソルON |
| G_CLR_ON | **0x90** | なし | **グラフィック画面表示ON（必須）** |
| GPALET | **0x94** | D1=パレット番号, D2=色 | **グラフィックパレット設定** |

> **[よくある間違い]** GPALET の IOCS 番号を 0x11 とする情報があるが、正しくは **0x94**。

### スプライト制御系

| 関数名 | IOCS番号 (D0) | 引数 | 備考 |
| --- | --- | --- | --- |
| SP_INIT | 0xC0 | なし | スプライト初期化 |
| SP_ON | 0xC1 | なし | スプライト表示ON |
| SP_OFF | 0xC2 | なし | スプライト表示OFF |
| SP_CGCLR | 0xC3 | D1=PCG番号 | PCGパターークリア |
| SP_PALET | 0xCF | D1=0x80000000\|番号, D2=block, D3=色 | bit31=設定モード |

## 5. G_CLR_ON の必要性

GVRAM に描画した内容を画面に表示するには、**G_CLR_ON (IOCS 0x90) の呼び出しが必須**。

- G_CLR_ON なし: GVRAM に書き込んでも何も表示されない（スプライトのみ表示）
- G_CLR_ON あり: GVRAM の内容がグラフィックプレーンとして表示される

呼び出しタイミングは CRTMOD 後、メインループ前でよい。

## 6. CRTMOD モード一覧 (主要)

| モード番号 | 解像度 | 色数 | 水平周波数 | 備考 |
| --- | --- | --- | --- | --- |
| 9 | 768×512 | 16 | 31kHz | 高解像度 |
| 10 | 256×256 | 256 | 31kHz | **ゲーム向け。PX68k で動作確認済み** |
| 16 | 512×512 | 16 | 15kHz | デフォルト |

CRTMOD(10) は 256×256 ドット表示だが、VRAM ページは 512×512。
CRTC スクロールレジスタで 256×256 の表示ウィンドウを移動可能。

## 7. 256色モードの GVRAM フォーマット

CRTMOD(10) (256 色モード) における GVRAM アクセス:

- **1ドット = 1ワード (2バイト)** — 下位 8 ビットにパレットインデックス、上位 8 ビットは無視される
- **ストライド = 512ワード/ライン (1024バイト)** — 仮想画面が512x512の時は色モード・解像度に関わらず固定
- **GVRAM先頭 = $C00000**
- **総サイズ = 512×512×2 = 524,288バイト (512KB)** (1 ページ分)
- **座標計算**: `Address = $C00000 + (512 * Y + X) * 2`

```c
/* ピクセル (x, y) へのアクセス */
volatile uint16_t *p = (volatile uint16_t *)0xC00000;
p[y * 512 + x] = palette_index; /* 0-255, 上位8ビットは無視 */
```

> 仮想画面が512x512の時は**16/256/65536色モードすべてで1ドット=2バイト。** GVRAMハードウェアが512ワード固定幅で
> 構成されているため、色モードによらずストライドは1024バイト。
> 詳細は `01_メモリマップ.md` および `02_DMA.md` を参照。

### パレット色フォーマット

グラフィックパレット (GPALET) の色データ (D2) は RGB555 + Valid ビット:

```c
#define PAL(r,g,b) ((uint16_t)((((g)&0x1F)<<11)|(((r)&0x1F)<<6)|(((b)&0x1F)<<1)|1))
/* 例: 白 = PAL(31,31,31) = 0xFBE1 */
```

## 8. CRTC スクロールレジスタ

256 色モードでのハードウェアスクロール。表示ウィンドウ (256×256) を VRAM ページ (512×512) 上で移動する。

| レジスタ | アドレス | C言語アクセス | 機能 |
| --- | --- | --- | --- |
| Scroll 0 X | $E80018 | `CRTC[12]` | 水平スクロールオフセット (0-511) |
| Scroll 0 Y | $E8001A | `CRTC[13]` | 垂直スクロールオフセット (0-511) |

```c
#define CRTC ((volatile uint16_t *)0xE80000)

/* 水平スクロール（左にスクロール＝値を増加） */
scroll_x = (scroll_x + 1) & 511;
CRTC[12] = scroll_x;
```

値を増やすと表示ウィンドウが右に移動し、画面上のコンテンツが左に流れる。

## 9. VSync 待ち（MFP GPIP4 ポーリング）

最も確実で低オーバーヘッドな VSync 待ち手法。

```c
volatile char *mpf = (volatile char *)0xe88001;
while (!((*mpf) & 0x10)) ;  /* 表示期間開始を待つ */
while  ((*mpf) & 0x10) ;    /* 垂直帰線期間（ブランク）を待つ */
```

- `$E88001` は MFP の GPIP レジスタ
- Bit 4 = V-DISP 信号 (1=表示期間, 0=帰線期間)
- この順序で両エッジを待つことで 1 フレームを正確に同期

## 10. ビルド・実行手順

```bash
export PATH="/opt/homebrew/Cellar/elf2x68k/20260331/bin:$PATH"
make clean && make && make install
```

- `make install` は `cp g1.x ~/tmp` で PX68k の HDD イメージに配置
- PX68k の Human68k 上で `g1.x` を実行
