# X68000 SASI

## 1. SASIバスの物理構造と信号特性の解析

1987年に登場したX68000は、単なる「ホビーPC」の枠を超えた「パーソナルワークステーション」であった。その強力なアーキテクチャの根幹を支えたのが、標準装備されたSASI（Shugart Associates System Interface）である。PC-9801等の競合機がHDDを依然として高価なオプションとして扱っていた時代、本体設計の段階でストレージ拡張を前提としたバスを組み込んだことは、極めて戦略的な差別化であった。現代のストレージ規格の祖であるSCSIへと続くこのインターフェースを理解することは、X68000の低レイヤ設計思想を解読することと同義である。

システムアーキテクトの視点から見れば、SASIバスは極めて「ハードウェア的」な側面が強く、すべての信号は「負論理（Negative Logic / Lアクティブ）」で定義されている。

### 信号群の機能評価と物理ピンアサイン

Source Context（p.431-432）に基づき、バス信号の役割を物理レベルで整理する。

|   |   |   |
|---|---|---|
|信号名|フルスペル|役割と機能評価（すべてLアクティブ）|
|**BSY**|BUSY|バス使用中を示す。全フェーズを通じて保持される最重要信号。|
|**SEL**|SELECT|ターゲットの選択。イニシエータが特定のIDを叩き起こす際に使用。|
|**C/D**|CONTROL/DATA|バスの情報属性を定義。L=コマンド/ステータス、H=データ。|
|**I/O**|INPUT/OUTPUT|転送方向。L=ターゲットから本体（IN）、H=本体からターゲット（OUT）。|
|**REQ**|REQUEST|ターゲットによるデータ転送要求。非同期ハンドシェイクの起点。|
|**ACK**|ACKNOWLEDGE|REQに対するイニシエータの応答。非同期通信を成立させる要。|
|**RST**|RESET|全デバイスの強制初期化。物理的な電気リセット。|
|**MSG**|MESSAGE|通信完了等、バスの状態に関する情報の伝達に使用。|

### So What? レイヤー：オープンコレクタと非対称性の合理性

SASIバスの物理層において特筆すべきは、信号線が「オープンコレクタ（Open Collector）」構成であり、終端抵抗（ターミネータ）によるプルアップを前提としている点である。これにより、複数のデバイスが同時に「BSY」をドライブしても回路的な短絡（ショート）を招かず、ワイヤードオア接続によるバス・アービトレーションや競合監視を安全に行うことが可能となっている。

また、信号制御の「非対称性」も重要である。C/D、I/O、MSG、REQといったバスの状態を決定する主要信号はすべてターゲット側が制御権を握っている。イニシエータ（本体）はターゲットから提示された「フェーズ」に従う受動的な立場にあり、この設計がデバイスごとの速度差を吸収しつつ、バス全体の安定性を担保しているのである。

物理的な信号ラインの理解を踏まえ、次節ではこれらの信号が時間軸上でどのように組み合わさり、通信のフェーズを形成していくのかを解説する。

## 2. SASIバス・プロトコル：フェーズ遷移の論理構造

SASIにおける通信は、単なるデータの羅列ではなく、厳格な「フェーズ遷移」によって統制されている。この状態遷移モデルの採用は、信頼性の低い外部接続環境下において通信の整合性を維持するための高度な戦略である。

### フェーズ遷移の論理ステップ

Source Context（p.433-438）に基づき、バス動作を分解すると以下の順序を辿る。

1. **バスフリー（BUS FREE）**：BSY/SELが「H」であり、バスが開放されている静止状態。
2. **セレクション（SELECTION）**：イニシエータがデータバスにターゲットIDを出力し（後述）、SELを「L」にして特定のHDDを選択する。
3. **コマンド（COMMAND）**：ターゲットがC/D=L、I/O=Hとし、本体から6バイトのCDBを受け取る。
4. **データ（DATA）**：I/Oの極性に基づき、実データのIN/OUTが実行される。
5. **ステータス（STATUS）**：ターゲットが実行結果（1バイト）を本体へ送出。
6. **メッセージ（MESSAGE）**：完了通知（$00）を送信し、バスを開放する。

### REQ/ACKハンドシェイクの分析

データ転送の確実性を担保するのが「非同期ハンドシェイク（Asynchronous Handshake）」である。ターゲットがREQを立て、イニシエータがACKで応える。この一対一の応答確認は、共通クロックを必要としないため、ドライブ側のコントローラの処理速度に依存せず、正確なデータ同期を実現している。

### So What? レイヤー：セレクション時のIDマッピングと拡張性の限界

セレクションフェーズでは、データバスの8ビットがそのままIDとしてマッピングされる（例：ID #0ならBit 0を、ID #1ならBit 1を「L」にする）。この仕組みにより、物理的なスイッチ設定のみで最大8台までのデバイス識別を可能にしているが、同時に「バス幅＝最大デバイス数」というアーキテクチャ上の制約も生んでいる。このIDマッピングによる競合回避策は、簡潔ながらも当時のストレージエコシステムを規定する強固な枠組みであった。

プロトコルの論理的フローを把握したところで、これらをソフトウェアから制御するための具体的なハードウェア窓口であるI/Oポートの設定を確認する。

## 3. X68000ハードウェアにおけるSASI I/Oポートの配置と定義

68000アーキテクチャにおいて、I/O操作はすべて「メモリマップドI/O（MMIO）」として抽象化されている。CPUは特定のメモリ番地に対して読み書きを行うだけで、物理的なSASI信号を直接操作できる。

### I/Oアドレスマップの詳細定義

Source Context（p.439）およびハードウェア仕様に基づき、E96001からE9600Fまでのレジスタを定義する。

- **$E96001：データレジスタ（Data Register）**
    - バス上の8ビットデータに直接アクセス。
- **$E96003：ステータス [Read] / リセット [Write]**
    - Read：主要信号（BSY, REQ, I/O, C/D, MSG）のリアルタイム監視。
    - Write：任意の値の書き込みでバスリセット（RST）を発行。
- **$E96005：セレクト [Write] / セレクトコード [Read]**
    - セレクション動作のトリガー。
- **$E96009：信号状態監視（BSY/REQ）**
    - Bit 7: BSY, Bit 6: REQ の状態を保持。ポーリング制御の最前線となる。
- **$E9600B：相関信号監視（I/O, C/D, MSG）**
    - Bit 7: I/O, Bit 6: C/D, Bit 5: MSG。現在のフェーズを判別するために使用。
- **$E9600D：一般ステータス**
    - インターフェース内部の状態やエラーの有無を確認。
- **$E9600F：割り込み制御（Interrupt Control）**
    - 転送完了後の割り込みベクタ生成や許可を管理。

### So What? レイヤー：MMIOによるMOVE命令の効率化

x86アーキテクチャのような専用のIN/OUT命令を必要とせず、68000の強力なMOVE命令、あるいはポインタ操作でI/Oを叩ける点は、ドライバ開発において圧倒的な優位性を持つ。特にアドレス間接参照を用いたループ処理（`MOVE.B (A0), (A1)+`など）は、SASIのデータ転送において最小のオーバーヘッドで最大の効率を叩き出すことができたのである。ただし、一つの番地が読み書きで全く異なる意味（例：$E96003のStatus/Reset）を持つため、ビット演算時の「Read-Modify-Write」動作には細心の注意が必要となる。

ハードウェアのアクセスポイントが確定したことで、いよいよディスク装置に対して発行する「コマンド」の構造と実行手順へと踏み込む。

## 4. SASIコマンドセットとコントロールブロックの詳細

SASIの核心は「コマンド・ディスクリプタ・ブロック（CDB）」による抽象化である。CPUはHDDの磁気ヘッドの物理座標を意識することなく、論理的な指示を発行する。

### コマンドフォーマット（6バイトCDB）の解剖

Source Context（p.440-445）によれば、コマンドは以下の6バイトで構成される。

1. **第0バイト**：オペレーションコード（READ=08, WRITE=0A等）
2. **第1バイト**：LUN（Logical Unit Number）およびLBAの上位5ビット
3. **第2〜3バイト**：LBA（論理ブロックアドレス）の残りの16ビット
4. **第4バイト**：転送ブロック数（0の場合は256ブロックを指す場合がある）
5. **第5バイト**：コントロールバイト（通常は$00）

### 主要コマンドの機能評価

- **TEST UNIT READY ($00)**：ドライブの稼働準備確認。
- **REQUEST SENSE ($03)**：エラー発生時の詳細ステータス（センスキー）取得。
- **READ (****08) / WRITE (****0A)**：LBA指定によるデータ転送。
- **SEEK ($0B)**：ヘッドを目標ブロックへ事前移動させる。

### So What? レイヤー：21ビットLBAの壁とSCSIへの継承

SASIの6バイトCDBにおけるLBAフィールドは実質21ビットに限られていた。1ブロック512バイト換算で、最大1GB（あるいは実装により2GB）という容量の壁がここで規定されていたのである。1987年当時には「広大」に見えたこの空間も、技術の進歩とともにSCSI-2の10バイトCDBへと進化せざるを得なかった。しかし、この論理ブロックという概念こそが、ハードウェアとOSを分離し、ストレージの標準化を成し遂げた最大の功績である。

コマンドの論理的な構造を理解した後は、これを実際のC言語プログラムとしてどのように実装し、ハードウェアを駆動させるべきかを具体的に検討する。

## 5. 実装分析：サンプルプログラムに見る低レイヤ制御の作法

低レイヤ制御において、仕様書通りの挙動を保証するのはコード側の「監視と待機」である。

### SASIディスク読み出しのアルゴリズム

Source Context（p.446-452）に基づき、典型的な実行フローを再現する。

1. **特権モードへの移行**：`SUPER()`を呼び出し、MMIO領域へのアクセス権を確保。
2. **ターゲット・セレクション**：データレジスタにIDビットをセットし、セレクションポートを叩く。
3. **フェーズ待機とコマンド送出**：C/D、I/O信号を監視（ポーリング）し、コマンドフェーズ移行を確認。6バイトのCDBを順次書き込む。
4. **データ転送ループ**：REQ信号の立ち上がりを確認し、ACKを返しながらデータレジスタから1バイトずつ取得。
5. **終了処理**：ステータスおよびメッセージを受け取り、BSYの解除を待ってバスを解放。

### So What? レイヤー：`volatile`キーワードとポーリングの必然性

現代的なコンパイラ（GCC等）を使用する場合、I/Oレジスタを指すポインタには必ず `volatile` キーワードを付与しなければならない。さもなくば、コンパイラは「ループ内で値が変化しない」と誤判断し、REQやBSYの監視ループ（いわゆる「ぐるぐる回る」ウェイト処理）を最適化によって削除してしまうからだ。当時のXCコンパイラとGCCでの挙動の差は、このキーワードひとつで決まる。

また、割り込みを使用せずポーリングを採用する手法は、一見非効率だが、CPUとHDDのタイミングが極めてシビアなSASI制御においては、不確定な割り込みレイテンシを排除し、決定論的な動作を保証するための賢明な選択であった。

## 6. 総括：高度なSASI制御のための設計指針

物理層の「負論理・オープンコレクタ」から、プロトコル層の「非対称フェーズ遷移」、そして実装層の「MMIOと`volatile`」に至る垂直統合的な理解こそが、X68000という名機の心臓部を掌握するための鍵である。

### 制御のチェックリスト：5つの核心項目

1. **バス終端の確認**：オープンコレクタバスにおいて、ターミネータのない環境は波形の乱れと転送エラーを直結させる。
2. **負論理の徹底**：信号の「真」は電圧が「低い」状態であることを常に意識せよ。
3. **ハンドシェイクの完遂**：REQに対してACKを返さない、あるいは中途半端なタイミングでの解放はターゲットをハングアップさせる。
4. `**volatile**`**による最適化抑制**：C言語によるドライバ記述において、これは「作法」ではなく「義務」である。
5. **21ビットLBAの境界認識**：SASI規格の限界を超えたアクセスは、データの破壊を招く。

### トラブルシューティングの要諦

通信エラー時は、まず「どの信号が、どのフェーズで固まったか」をI/Oポート$E96009/Bから読み取ること。BSYが落ちないならターゲット側のハング、REQが来ないならフェーズ不整合。物理信号と論理フェーズを対照させることで、原因は自ずと明らかになる。

本ドキュメントが提供する体系的知識が、X68000というプラットフォームの深層を理解し、その技術的遺産を現代に継承するための不朽の道標となれば幸いである。
