# X68000 SASI

## 1. SASIバスの物理構造と信号特性の解析

1987年に登場したX68000は、単なる「ホビーPC」の枠を超えた「パーソナルワークステーション」であった。その強力なアーキテクチャの根幹を支えたのが、標準装備されたSASI（Shugart Associates System Interface）である。PC-9801等の競合機がHDDを依然として高価なオプションとして扱っていた時代、本体設計の段階でストレージ拡張を前提としたバスを組み込んだことは、極めて戦略的な差別化であった。現代のストレージ規格の祖であるSCSIへと続くこのインターフェースを理解することは、X68000の低レイヤ設計思想を解読することと同義である。

システムアーキテクトの視点から見れば、SASIバスは極めて「ハードウェア的」な側面が強く、すべての信号は「負論理（Negative Logic / Lアクティブ）」で定義されている。

### 信号群の機能評価と物理ピンアサイン

Source Context（p.431-451）に基づき、バス信号の役割を物理レベルで整理する。

|   |   |   |
| :--- | :--- | :--- |
|信号名|フルスペル|役割と機能評価（すべてLアクティブ）|
|**BSY**|BUSY|バス使用中を示す。全フェーズを通じて保持される最重要信号。|
|**SEL**|SELECT|ターゲットの選択。イニシエータが特定のIDを叩き起こす際に使用。|
|**C/D**|CONTROL/DATA|バスの情報属性を定義。L=コマンド/ステータス、H=データ。|
|**I/O**|INPUT/OUTPUT|転送方向。L=ターゲットから本体（IN）、H=本体からターゲット（OUT）。|
|**REQ**|REQUEST|ターゲットによるデータ転送要求。非同期ハンドシェイクの起点。|
|**ACK**|ACKNOWLEDGE|REQに対するイニシエータの応答。非同期通信を成立させる要。|
|**RST**|RESET|全デバイスの強制初期化。物理的な電気リセット。|
|**MSG**|MESSAGE|通信完了等、バスの状態に関する情報の伝達に使用。|

### So What? レイヤー：オープンコレクタと非対称性の合理性

SASIバスの物理層において特筆すべきは、信号線が「オープンコレクタ（Open Collector）」構成であり、終端抵抗（ターミネータ）によるプルアップを前提としている点である。これにより、複数のデバイスが同時に「BSY」をドライブしても回路的な短絡（ショート）を招かず、ワイヤードオア接続によるバス・アービトレーションや競合監視を安全に行うことが可能となっている。

SASIバスでは、バス上に最大8つのコントローラが接続でき、それぞれ固有のID（0〜7）が振られる。さらに各コントローラの下に最大8台のデバイス（論理ユニット）が接続できるが、実際にはほとんどのコントローラでLUN 0と1しかサポートされておらず、Human68kもそれに準じるため、実際に使用できるのは最大16台まで。内蔵ディスクがIDを1つ使用するため、外接可能なのは14台、内蔵と合わせて15台までとなる。

また、信号制御の「非対称性」も重要である。C/D、I/O、MSG、REQといったバスの状態を決定する主要信号はすべてターゲット側が制御権を握っている。イニシエータ（本体）はターゲットから提示された「フェーズ」に従う受動的な立場にあり、この設計がデバイスごとの速度差を吸収しつつ、バス全体の安定性を担保しているのである。

物理的な信号ラインの理解を踏まえ、次節ではこれらの信号が時間軸上でどのように組み合わさり、通信のフェーズを形成していくのかを解説する。

## 2. SASIバス・プロトコル：フェーズ遷移の論理構造

SASIにおける通信は、単なるデータの羅列ではなく、厳格な「フェーズ遷移」によって統制されている。この状態遷移モデルの採用は、信頼性の低い外部接続環境下において通信の整合性を維持するための高度な戦略である。

### フェーズ遷移の論理ステップ

Source Context（p.433-445）に基づき、バス動作を分解すると以下の順序を辿る。

1. **バスフリー（BUS FREE）**：BSY/SELが「H」であり、バスが開放されている静止状態。すべての信号が'0'（非アクティブ）。
2. **セレクション（SELECTION）**：ホストがデータバスにターゲットIDを出力し（ID#0=$01, ID#3=$08等）、SELを'1'（Lレベル）にして特定のHDDを選択する。選択されたコントローラがBSY='1'で応答。X68000ではセレクション用ポート（$E96007）への書き込みで自動的にSELが動作する。
3. **コマンド（COMMAND）**：I/O='0', C/D='1', MSG='0'。ターゲットが6バイトのCDBをREQ-ACKハンドシェイクで受け取る。X68000ではデータポート（$E96001）へのアクセスで自動的にハンドシェイクが行われる。
4. **データ（DATA）**：C/D='0', MSG='0'。WRITE時I/O='0'、READ時I/O='1'。REQ-ACKハンドシェイクで転送。DMA転送も可能。
5. **ステータス（STATUS）**：I/O='1', C/D='1', MSG='0'。ターゲットが実行結果（1バイト、正常=$00）を本体へ送出。
6. **メッセージ（MESSAGE）**：I/O='1', C/D='1', MSG='1'。完了通知（$00=コマンドコンプリート）を送信し、BSYを解除してバスを開放。

### REQ/ACKハンドシェイクの分析

データ転送の確実性を担保するのが「非同期ハンドシェイク（Asynchronous Handshake）」である。ターゲットがREQを立て、イニシエータがACKで応える。この一対一の応答確認は、共通クロックを必要としないため、ドライブ側のコントローラの処理速度に依存せず、正確なデータ同期を実現している。

### So What? レイヤー：セレクション時のIDマッピングと拡張性の限界

セレクションフェーズでは、データバスの8ビットがそのままIDとしてマッピングされる（ID #0なら$01、ID #3なら$08をバス上に出力）。この仕組みにより、物理的なスイッチ設定のみで最大8台までのデバイス識別を可能にしているが、同時に「バス幅＝最大デバイス数」というアーキテクチャ上の制約も生んでいる。セレクションが成功すると、選択されたコントローラがBSY='1'で応答し、ホストはSELを'0'に戻してフェーズを終了する。BSYが応答しない場合はコントローラが存在しないものとみなしてエラー処理を行う。

プロトコルの論理的フローを把握したところで、これらをソフトウェアから制御するための具体的なハードウェア窓口であるI/Oポートの設定を確認する。

## 3. X68000ハードウェアにおけるSASI I/Oポートの配置と定義

68000アーキテクチャにおいて、I/O操作はすべて「メモリマップドI/O（MMIO）」として抽象化されている。CPUは特定のメモリ番地に対して読み書きを行うだけで、物理的なSASI信号を直接操作できる。

### I/Oアドレスマップの詳細定義

Source Context（p.439）およびハードウェア仕様に基づき、SASIインタフェースポートを定義する。

- **$E96001：データレジスタ（Data Register）[R/W]**
  - バス上の8ビットデータに直接アクセス。**このポートへのアクセス時にハードウェアが自動的にREQ-ACKハンドシェイクを行う**ため、ソフトウェアでREQ/ACK信号を制御する手間を省ける。DMAチャンネルを使った転送も可能。
  - Read時： additionally, ステータスビットも取得可能（bit6=MSG, bit5=C/D, bit4=I/O, bit3=BSY, bit2=REQ）。

- **$E96003：ステータス/コントロールレジスタ**
  - **Read**：SASI制御信号の状態を確認（現在のフェーズ判別に使用）。
  - **Write**：SEL信号を'0'（Hレベル）にセットし、SASIバスに出力するデータをバスに載せる。通常$00を書き込む。セレクションフェーズ用ポート。

- **$E96005：リセットレジスタ [Write]**
  - SASIバスのRESET信号を一定期間'1'（Lレベル）にする。書き込みを行うと、SASIバスに接続されたハードディスクはすべてリセットされる。**リード/ライト中であっても強制的にリセットされる**ため、使用にあたっては十分注意。

- **$E96007：セレクトレジスタ [Write]**
  - DATAをSASIバスに出力するとともに、SEL信号を'1'（Lレベル）にセットする。セレクションフェーズでIDビットをバスに載せる際に使用。

### So What? レイヤー：MMIOによるMOVE命令の効率化

x86アーキテクチャのような専用のIN/OUT命令を必要とせず、68000の強力なMOVE命令、あるいはポインタ操作でI/Oを叩ける点は、ドライバ開発において圧倒的な優位性を持つ。特にアドレス間接参照を用いたループ処理（`MOVE.B (A0), (A1)+`など）は、SASIのデータ転送において最小のオーバーヘッドで最大の効率を叩き出すことができた。さらに、$E96001へのアクセス時にハードウェアが自動REQ-ACKハンドシェイクを行うため、ソフトウェアは単にデータポートを読み書きするだけでデータ転送が完了する。DMA転送も可能であり、DMAリクエスト信号はREQ='1'でアクティブ、ACK='1'で復帰する。ただし、一つの番地が読み書きで全く異なる意味（例：$E96003のRead=Status / Write=SEL Off）を持つため、ビット演算時の「Read-Modify-Write」動作には細心の注意が必要となる。

ハードウェアのアクセスポイントが確定したことで、いよいよディスク装置に対して発行する「コマンド」の構造と実行手順へと踏み込む。

## 4. SASIコマンドセットとコントロールブロックの詳細

SASIの核心は「コマンド・ディスクリプタ・ブロック（CDB）」による抽象化である。CPUはHDDの磁気ヘッドの物理座標を意識することなく、論理的な指示を発行する。

### コマンドフォーマット（6バイトCDB）の解剖

Source Context（p.440-445）によれば、コマンドは以下の6バイトで構成される。

1. **第0バイト**：オペレーションコード（上位3ビットでクラス分類、通常はクラス0=$00。READ=$08, WRITE=$0A等）
2. **第1バイト**：LUN（Logical Unit Number、3ビット）およびLBAの上位5ビット
3. **第2〜3バイト**：LBA（論理ブロックアドレス）の残りの16ビット
4. **第4バイト**：転送ブロック数（0の場合は256ブロックを指す）
5. **第5バイト**：コントロールバイト（通常は$00。SCSIではフォワーダ等に使用）

### 主要コマンドの機能評価

- **TEST DRIVE READY ($00)**：ドライブの稼働準備確認。データ転送なし、ステータスフェーズに直接移行。
- **RECALIBRATE ($01)**：ヘッドをトラック0に戻す。ヘッド位置ずれ時の復帰手段。
- **REQUEST SENSE STATUS ($03)**：エラー発生時の詳細ステータス（センスキー）取得。4バイトのステータスをデータ転送フェーズで引き取る。
- **FORMAT DRIVE ($04)**：ドライブの物理フォーマット。コマンド発行後の処理はすべてコントローラ側で実行。
- **READ ($08) / WRITE ($0A)**：LBA指定によるデータ転送。X68000では1ブロック256バイト。

### So What? レイヤー：21ビットLBAの壁とSCSIへの継承

SASIの6バイトCDBにおけるLBAフィールドは実質21ビットに限られていた。X68000では1ブロック256バイト換算で、2^21×256＝512MBという容量の壁がここで規定されていたのである。1987年当時には「広大」に見えたこの空間も、技術の進歩とともにSCSI-2の10バイトCDBへと進化せざるを得なかった。しかし、この論理ブロックという概念こそが、ハードウェアとOSを分離し、ストレージの標準化を成し遂げた最大の功績である。

コマンドの論理的な構造を理解した後は、これを実際のC言語プログラムとしてどのように実装し、ハードウェアを駆動させるべきかを具体的に検討する。

## 5. 実装分析：サンプルプログラムに見る低レイヤ制御の作法

低レイヤ制御において、仕様書通りの挙動を保証するのはコード側の「監視と待機」である。

### SASIディスク読み出しのアルゴリズム

Source Context（p.446-452）に基づき、典型的な実行フローを再現する。

1. **特権モードへの移行**：`SUPER()`を呼び出し、MMIO領域へのアクセス権を確保。
2. **ターゲット・セレクション**：データレジスタにIDビットをセットし、セレクションポート（$E96007）を叩く。自動的にSEL信号が動作する。
3. **フェーズ待機とコマンド送出**：$E96003をポーリングし、コマンドフェーズ移行を確認。$E96001に6バイトのCDBを書き込むと、ハードウェアが自動REQ-ACKハンドシェイクを行う。
4. **データ転送ループ**：$E96001へのアクセスで自動ハンドシェイクが行われる。DMA転送も可能。DMAモードは'11'（最初の1バイトだけがオートリクエスト、残りは外部転送要求）を設定し、最初の1バイト目はREQ='1'を確認してからDMAを開始する。
5. **終了処理**：ステータスおよびメッセージを$E96001から受け取り、BSYの解除を待ってバスを解放。

### So What? レイヤー：`volatile`キーワードとポーリングの必然性

現代的なコンパイラ（GCC等）を使用する場合、I/Oレジスタを指すポインタには必ず `volatile` キーワードを付与しなければならない。さもなくば、コンパイラは「ループ内で値が変化しない」と誤判断し、REQやBSYの監視ループ（いわゆる「ぐるぐる回る」ウェイト処理）を最適化によって削除してしまうからだ。

XCコンパイラでは`volatile`がサポートされていないため、`#define volatile`として無効にする必要がある。当時のXCコンパイラとGCCでの挙動の差は、このキーワードひとつで決まる。実際のサンプルプログラムでは、`#define volatile`をファイル先頭に入れてXCでのコンパイルを可能にしている。

また、割り込みを使用せずポーリングを採用する手法は、一見非効率だが、CPUとHDDのタイミングが極めてシビアなSASI制御においては、不確定な割り込みレイテンシを排除し、決定論的な動作を保証するための賢明な選択であった。

## 6. 総括：高度なSASI制御のための設計指針

物理層の「負論理・オープンコレクタ」から、プロトコル層の「非対称フェーズ遷移」、そして実装層の「MMIOと`volatile`」に至る垂直統合的な理解こそが、X68000という名機の心臓部を掌握するための鍵である。

### 制御のチェックリスト：5つの核心項目

1. **バス終端の確認**：オープンコレクタバスにおいて、ターミネータのない環境は波形の乱れと転送エラーを直結させる。
2. **負論理の徹底**：信号の「真」は電圧が「低い」状態であることを常に意識せよ。
3. **ハンドシェイクの完遂**：REQに対してACKを返さない、あるいは中途半端なタイミングでの解放はターゲットをハングアップさせる。
4. `**volatile**`**による最適化抑制**：C言語によるドライバ記述において、これは「作法」ではなく「義務」である。
5. **21ビットLBAの境界認識**：SASI規格の限界を超えたアクセスは、データの破壊を招く。

### トラブルシューティングの要諦

通信エラー時は、まず「どの信号が、どのフェーズで固まったか」をI/Oポート$E96003から読み取ること。BSYが落ちないならターゲット側のハング、REQが来ないならフェーズ不整合。物理信号と論理フェーズを対照させることで、原因は自ずと明らかになる。

本ドキュメントが提供する体系的知識が、X68000というプラットフォームの深層を理解し、その技術的遺産を現代に継承するための不朽の道標となれば幸いである。
