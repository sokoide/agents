# X68000 ジョイスティック

## 1. ジョイスティック・インタフェースの概要と戦略的役割

X68000が1987年に「パーソナルワークステーション」として登場した際、我々開発チームが意図したのは、単なるホビーマシンの枠を超えた、個人の直感や発想をリアルタイムに反映できるプラットフォームの構築であった。ジョイスティック・インタフェースは、その思想を具現化する重要な入力系の一つである。

### インタフェースの物理的構成とシリーズにおける一貫性

X68000は標準で2系統（ポート#1/#2）のジョイスティック端子を装備している。物理的にはアタリ規格準拠のD-Sub 9ピンを採用しているが、これは当時のデファクトスタンダードとの互換性を維持しつつ、独自拡張を可能にするための選択であった。 このインタフェース仕様は、初代機からACE、EXPERT、SUPER、そしてクロック向上を果たしたXVIに至るまで、ハードウェアの集積化が進む中でも完全な互換性が保たれている。これは、本機が「系譜」を通じてソフトウェア資産を継承し続けるための鉄則であった。

### システムI/O領域における低レイテンシ設計

技術的な核心は、ジョイスティック・レジスタがメモリマップ上の「システムI/O領域（E80000〜E9FFFF）」、具体的には **$E9A000** 以降に配置されている点にある。 本機にはOS（Human68k）レベルで `IOCS` 等のAPIが用意されているが、極限のレスポンスを求めるアクションゲームやリアルタイム・アプリケーションにおいては、CPUが直接このI/Oポートを叩く「ダイレクトI/Oアクセス」が推奨される。OSを介在させないポーリング処理により、1フレーム（16.7ms）以下の遅延を許容しない、ワークステーションの名に恥じない操作感を実現している。

## 2. I/Oポートおよび関連レジスタの詳細仕様

プログラマがジョイスティック入力を制御する上で、操作すべきレジスタのアドレスとビット構造を以下に定義する。

### ジョイスティックポートデータ・レジスタ

- **ポート#1 データ: $E9A001**
- **ポート#2 データ: $E9A003**

各ポートのデータは、68000のバス仕様に基づき奇数アドレスに配置されている。読み出されるデータは「負論理（スイッチONで'0'）」であり、以下のビットマップを持つ。

- **Bit 0:** 上 (UP)
- **Bit 1:** 下 (DOWN)
- **Bit 2:** 左 (LEFT)
- **Bit 3:** 右 (RIGHT)
- **Bit 4:** ボタンA (TRIGGER 1)
- **Bit 5:** ボタンB (TRIGGER 2)

### ジョイスティックコントロール・レジスタ ($E9A005)

単なるアタリ規格互換を超えた「So What?」の鍵を握るのが、このコントロール・レジスタである。本レジスタは、ポートのピン出力を制御するためのものであり、後述する多ボタンパッドやトラックマウス（トラックボール）などの拡張デバイスを識別・サンプリングするための「プロトコル制御」を司る。 MFP (MC68901) や I/O Controller LSI と同様に、ここを適切に叩くことで、X68000特有のデスクトップ・メディアとしての多様性が発揮されるのである。

## 3. コントロールワードの運用プロトコルと制御ロジック

X68000のジョイスティックポートは単なる入力端子ではなく、ソフトウェアからの「コントロールワード」送出によって信号線の役割を動的に変更できる双方向性を持っている。

### コントロールワードによるマルチプレクス

特定のピンを出力として運用し、デバイスに対して状態の切り替えを促す。これにより、物理的なピン数の制限をソフトウェア的な多重化（マルチプレクス）で克服している。標準の2ボタン・スティック以外の拡張デバイスを接続した際、このコントロールワードによるサンプリングを行わなければ、システムはそれらを認識することすらできない。

### CPUクロック（10MHz/16MHz）に依存するタイミング制約

ここでの実装において、システムプログラマはタイミング制御に極めて厳格でなければならない。 10MHz、あるいはXVIの16MHzで動作する68000 CPUにとって、ハードウェア信号のラッチや立ち上がり待ちは瞬きに等しい時間である。コントロールワードをレジスタに書き込んだ直後にポートを読み取っても、信号は電気的に安定していない。

- **ウェイトの必要性:** 信号の確定には数マイクロ秒の猶予が必要であり、CPUクロックサイクルに基づいた適切なNOP命令の羅列、あるいはソフトウェア・ループによるウェイトが不可欠である。この「制御の作法」を無視したコードは、XVIなどの高速機において深刻な誤動作を引き起こす。

## 4. プログラミング実践：実装例とデータ取得の最適化

本機の設計思想である「ステータスチェック（ポーリング）による入力検知」に基づいた、実戦的な実装例を提示する。

### 最適化された入力取得ルーチン (C言語例)

以下は、負論理を正論理に反転させ、特定のビットを抽出するプロフェッショナルな実装の一例である。

```c
/* ジョイスティック入力サンプリング（gcc環境想定） */
void sample_joystick() {
    /* 
     * volatile指定によりコンパイラの最適化による読み飛ばしを防止
     * アドレスは$E9A001を直接参照
     */
    volatile unsigned char *joy1 = (unsigned char *)0xE9A001;
    unsigned char raw_data;
    unsigned char active_data;

    raw_data = *joy1;
    active_data = ~raw_data; // 負論理を正論理へ変換

    // ボタンA (Bit 5) の判定: 0x20でマスク
    if (active_data & 0x20) {
        /* ボタンA押下時の処理 */
    }

    // 方向入力（上下左右）の判定
    if (active_data & 0x01) { /* 上 */ }
    if (active_data & 0x02) { /* 下 */ }
}
```

### 実装上のインパクト：チャタリングと優先順位

1. **チャタリング防止の必然性:** 16MHzで動作するCPUは、人間の指がスイッチを押し下げる間に発生する物理的な接点ノイズ（チャタリング）を数回〜数十回にわたってサンプリングしてしまう。ソフトウェア側で前回値との比較を行い、一定期間の安定を確認しなければ、メニューが二重に送られる等の不具合を生む。
2. **論理的優先順位付け:** ハードウェアの故障や特殊な操作により、上（Bit 0）と下（Bit 1）が同時に'0'になる等の異常入力が発生しうる。これをそのまま処理すれば、グラフィック上のキャラクタ座標が破壊される可能性がある。プログラム側で「上優先」などの論理規定を設けることが、堅牢なシステム構築の第一歩となる。

### 他コンポーネントとの連動戦略

取得した入力データは、即座にスプライトレジスタ（EB0000）や、グラフィックVRAM（C00000）の座標更新に反映させる。特にDMAC（HD63450）を用いた矩形領域転送（Source Context 61ページ参照）と組み合わせることで、ジョイスティックの入力に応じた高速かつ滑らかな画面スクロールが実現可能となる。これこそが、ハードウェアを直接制御するX68000プログラミングの醍醐味である。
