# 数値演算プロセッサ（MC68881）

X68000シリーズにおける数値演算プロセッサ（FPU）の搭載は、科学技術計算や3Dレンダリング、複雑な物理シミュレーションにおいて決定的な優位性をもたらします。本機に採用されたMC68881は、本来MC68020のコプロセッサとして設計されたものですが、X68000環境ではホストCPU（MC68000）の制約により「周辺I/Oデバイス」としてマッピングされています。本ドキュメントでは、システムアーキテクトの視点から、この変則的な接続形態におけるMC68881の完全制御手法を詳解します。

--------------------------------------------------------------------------------

## 1. 数値演算プロセッサの概要とシステム上の位置付け

X68000において、MC68881は物理的・論理的に特異な位置付けにあります。MC68020以降のCPUであれば「コプロセッサ・インタフェース」によりハードウェアが透過的に制御しますが、X68000ではアーキテクトが手動でバスサイクルを管理しなければなりません。

### 実装形態とアクセス戦略

- **XVI以前の機種：** 拡張ボード（CZ-6BP1等）を介して搭載。
- **XVI以降：** 本体基板上の専用ソケットに直接実装。
- **論理配置：** 標準ベースアドレスは `$E9E000`（2枚目は `$E9E080`）。この領域はシステムI/Oエリアに属するため、アクセスには必ず**スーパーバイザモード（SUPER(0)）**への移行が必要です。

### プログラミングの複雑性

68000はFPU専用命令（F-Line命令）を直接実行できないため、ソフトウェア側でFPUをI/Oとして叩き、68020のマイクロコードが行う「コプロセッサ・プロトコル」をシミュレートする必要があります。この手動制御が、高速演算の恩恵と実装の難易度をトレードオフにしています。

--------------------------------------------------------------------------------

## 2. MC68881の内部アーキテクチャとレジスタ構成

FPU制御の第一歩は、その内部レジスタ群の役割を完全に把握することです。これらはホストメモリには直接現れず、CIR（コプロセッサ・インタフェース・レジスタ）という窓口を通じて操作します。

### 浮動小数点データレジスタ（FP0～FP7）

- **構造：** 80ビット長の独立したレジスタが8本。
- **特性：** 内部演算は常にこの80ビット拡張精度で行われ、演算精度を最大限に維持します。

### 制御・ステータスレジスタ

1. **FPCR（コントロール）：**
    - **PREC（丸め精度）：** 単精度(32bit)、倍精度(64bit)、拡張精度(80bit)を選択。
    - **RND（丸めモード）：** 近似、ゼロ方向、正負の無限大方向を設定。
2. **FPSR（ステータス）：**
    - **コンディションコード：** 演算結果が負（N）、ゼロ（Z）、無限大（I）、NAN（非数）であるかを示す。
    - **商バイト：** FMOD/FREM命令実行時の計算結果の一部。
    - **例外ステータス/アクルード例外：** 5つのIEEE定義ビット（IOP:無効操作, OVFL:溢れ, UNFL:不足, DZ:零除算, INEX:不正確）を管理。アクルード例外はこれまでの演算結果の累積（OR）を保持するため、一連の計算後のバルクチェックに適しています。
3. **FPIAR（命令アドレス）：** 最後に実行された命令のメモリアドレスを保持。

## 3. データフォーマットと高精度演算のメカニズム

MC68881はIEEE 754に準拠した7つのデータ型をサポートします。

### サポートデータフォーマット一覧

|   |   |   |   |
|---|---|---|---|
|型|略称|サイズ|内部処理の挙動|
|バイト整数|B|8bit|転送時に拡張精度へ変換|
|ワード整数|W|16bit|転送時に拡張精度へ変換|
|ロングワード整数|L|32bit|転送時に拡張精度へ変換|
|単精度実数|S|32bit|指数8bit, 小数23bit|
|倍精度実数|D|64bit|指数11bit, 小数52bit|
|拡張精度実数|X|96bit|内部80bit。メモリ上では上部16bitがゼロ|
|パック型10進数|P|96bit|17桁の仮数を持つBCD形式|

### 内部演算の戦略的評価

68881の最大の特徴は、入力が何であれ**内部演算が常に80ビット拡張精度**で行われる点にあります。

- **利点：** 中間計算における精度落ちを極限まで排除。
- **オーバーヘッド：** 単精度等の外部データを扱う際、必ず変換プロセスが発生します。
- **特殊数：** 非正規化数、無限大、NANの定義により、ゼロ除算やオーバーフロー発生時も計算フローを即座に破綻させず、例外処理での事後回収を可能にしています。

## 4. コプロセッサ・インタフェース・レジスタ（CIR）の深層

X68000において68881を制御する唯一の手段が、以下の32バイトのI/Oブロックです。

### CIRメモリマップ（ベースアドレス: $E9E000 / $E9E080）

|   |   |   |   |
|---|---|---|---|
|相対オフセット|レジスタ名|属性|機能|
|$00|応答 (Response)|R|FPUの状態・要求（プリミティブ）を取得|
|$02|コントロール (Control)|W|アボート（実行中断）の指示|
|$04|セーブ (Save)|R|内部状態の読み出し（コンテキストスイッチ用）|
|$06|リストア (Restore)|R/W|内部状態の復旧|
|$0A|コマンド (Command)|W|演算命令（オペレーションワード）の発行|
|$0E|コンディション (Condition)|W|条件判定の要求|
|$10|オペランド (Operand)|R/W|実際のデータ転送ポート|
|$14|レジスタ選択 (Register Select)|R|FMOVEM時の対象レジスタリスト取得|
|$18|命令アドレス (Instruction Addr)|W|FPUにホストのPCを通知|
|$1C|オペランドアドレス (Operand Addr)|W|転送先アドレス（通常は未使用）|

## 5. 応答プリミティブによる通信プロトコルの制御

ホストCPUは「応答CIR」を読み取り、そのビットパターンに従ってFPUをサービシングします。

### 同期制御の核心：CA（Come Again）ビット

応答CIRの第15ビットは**CAビット**であり、これが「1」である限り、ホストは「Master Sync」ループを維持し、FPUの要求に応じ続ける必要があります。

### 主要プリミティブと例外ベクタ

FPUが返す「例外取得プリミティブ」には、下位8ビットに以下のベクタ番号が含まれます。

|   |   |   |
|---|---|---|
|ベクタ番号|エラー内容|発生条件|
|$0B|F-Line|未定義命令の発行|
|$31|Inexact|演算結果が丸めにより不正確|
|$32|Zero Div|ゼロによる除算|
|$33|Underflow|指数部が表現範囲を下回った|
|$34|Operand Error|無効な演算対象（NAN等）|
|$35|Overflow|指数部が表現範囲を超えた|
|$36|Signaling NAN|NANを演算対象にした|

## 6. 命令実行シーケンスとオペレーション・クラス別の制御フロー

### 1. 内部レジスタ間演算（OPクラス 000）

- コマンドCIRに命令を書き込む。
- 応答CIRをループで読み、値が `$0900`（内部処理中）から `$0802`（アイドル）になるのを待機。

### 2. 外部データからレジスタへの転送・演算（OPクラス 010）

- コマンドCIRに命令を書き込む。
- 応答CIRが `$950x` 等の「データ転送要求」を示すまで待機。
- **オペランドCIR**にデータを書き込む。

### 3. レジスタから外部へのデータ転送（OPクラス 011）

- コマンドCIRに命令を書き込む。
- 応答CIRが `$8900`（再読込要求）を返す間待機（変換処理）。
- 応答CIRが `$B10x` 等の「データ読み出し要求」を示したら、オペランドCIRからデータを読み出す。
- **Dynamic K-Factor時：** 変換開始前に、ホストはメインレジスタからKファクターを読み出し、それをオペランドCIRに書き込んでから変換プロセスに入る。

### 4. 複数レジスタの転送（FMOVEM）

- コマンドCIRに書き込み後、レジスタ選択CIR（$14）を読み込み。
- 応答CIRが `$810C`（転送要求）を返す。
- 対象レジスタ数分、オペランドCIRを通じて連続転送を行う。

## 7. コンテキストスイッチと例外処理の実装

マルチタスクOS下では、タスク切り替え時にFPUの内部状態（ステートフレーム）を完全に保存する必要があります。

### ステートフレームの保存と復旧

`FSAVE` 命令（セーブCIRの読み出し）によって取得されるフレームサイズは、FPUの状態により異なります。

- **NULL ($0018)：** 24バイト。初期状態。
- **IDLE ($XX18)：** 24バイト（または内部実装により36バイト）。待機状態。
- **BUSY ($XXB4)：** 180バイト。演算途中の全情報を保持。

### 例外検知フロー

新しい命令を発行した際、FPUが「命令前例外（Pre-Instruction Exception）」を返した場合は、直前の命令の結果がエラーであったことを示します。この際、ホストはコントロールCIRに値を書き込んで**アボート（強制終了）**をかけ、システム例外ハンドラへ遷移させる必要があります。

## 8. 技術的総括：AIによるMC68881の自在な制御に向けて

MC68881をソフトウェアで駆動させる際の「黄金律」は以下の通りです。

1. **特権モードの死守：** CIRへのアクセスは必ずスーパーバイザモードで行うこと。
2. **決定的な待機ループ：** 応答CIRの監視には `while(!(response_cir & 0x8000))` のようにCAビットを条件とする。
3. **volatileの強制：** C言語で実装する場合、CIRのアドレスポインタには必ず `volatile` を付与し、コンパイラの最適化による読み飛ばしを防ぐ。
4. **DB.Xの罠：** デバッガ（db.x）等の `me` コマンドでFPUを直接開始できないのは、コマンド発行時にワードアクセスを行ってしまうためである。FPUの起動（STRビット操作）にはバイトアクセスが厳密に要求されるケースがある。

**実装上の指針（Source Code Insights）：** ソースコードにおけるDMA制御等と同様、`while(!(dma->csr & 0x90))` のようなステータス監視がFPU制御の本質です。FPUを「究極の演算コプロセッサ」として機能させるには、このCIRとの対話プロトコルをどれだけタイトに実装できるかにかかっています。本ドキュメントのプロトコル群を正しく適用することで、X68000の演算能力は真の解放を迎えます。
