# X68000 リアルタイムクロック（RTC）

本稿では、シャープ X68000シリーズにおけるリアルタイムクロック（RTC）のハードウェア仕様と、その低レイヤーにおける制御技法を詳解する。X68000が「パーソナルワークステーション」として定義される所以の一つである、高度な電源管理と時刻同期のメカニズムを、アーキテクチャの深層から紐解いていく。

--------------------------------------------------------------------------------

## 1. X68000システムにおけるRTCの戦略的役割と配置

X68000において、RTCは単なる「時計」ではない。それはシステムの自律的な起動（ALARM機能）や、精密なシステム管理を司る戦略的なペリフェラルである。

### システムI/O領域における物理配置

X68000のメモリマップにおいて、RTCはシステムI/O領域内の **E8A000** ブロックに割り当てられている（『Source Context』20ページ参照）。特筆すべきは、68000の16ビットデータバスにおける8ビットペリフェラルの接続方式である。X68000の設計ポリシーに基づき、RTCレジスタはデータバスの下位8ビット（D0-D7）に接続されているため、CPUからは ******E8A001** 以降の**奇数アドレス**として認識される。開発者はこのアドレスアライメントを厳守しなければならない。

### 電源制御とMFP（MC68901）との連携

RTCのALARM信号は、MFPの汎用I/Oポートである **GPIP0**（$E88001）へと直結されている（81ページ）。これは単なる入力信号ではなく、システムをスリープ状態から復帰させる「ALARM信号による自動起動」のトリガーとして機能する。この物理的接続こそが、X68000を無人運用が可能なワークステーションたらしめている要因である。

## 2. RTC周辺ブロック図と内部アーキテクチャの定義

RTC周辺のハードウェア境界を定義することは、確実な制御を行う上で不可欠である。

### ハードウェア信号の流儀

RTCから出力されるALARM信号は、MFP（MC68901）のGPIP0へと入力され、最終的にはCPUの割り込み、あるいは電源制御ロジックへと到達する。

- **MFPの役割:** ALARM信号はGPIP0に接続されているため、MFP内では「最も優先順位の低い割り込み要因」（83ページ）として扱われる。開発者は、ALARMをトリガーとしてソフトウェア処理を行う際、MFPのIERB（割り込みイネーブル）およびIMRB（割り込みマスク）の適切な設定に加え、AER（アクティブエッジレジスタ：$E88003）を操作して、信号の立ち下がり/立ち上がりのどちらで割り込みを発生させるかを明示的に定義しなければならない（82ページ）。
- **システムポートによる監視:** 電源投入要因の事後判定は、システムポート#2（**E89001**）等を通じて行われる（518ページ）。初期ドラフトで見られたE88xxx領域との混同は、MFPレジスタとの衝突を招く致命的な誤りであり、厳に慎まれるべきである。

## 3. RTC内部レジスタの詳細仕様と制御ビットの解析

RTCの内部レジスタは、ベースアドレス $E8A001 から奇数バイト間隔で配置されている。以下に、主要なレジスタとその戦略的意義を列挙する。

|   |   |   |
|---|---|---|
|オフセット|レジスタ名|機能概要|
|+01, +03...|時刻カウンタ群|秒、分、時、日、月、年（各桁ごと）|
|+$1B|アジャストレジスタ|水晶発振子の誤差を補正し、時刻精度を微調整する。|
|+$1D|12/24時間セレクタ|表記モードの切り替え。ソフトウェアの表示ロジックに影響する。|
|+$1F|閏年カウンタ|カレンダーの整合性をハードウェアレベルで維持する。|
|+$21|MODE レジスタ|動作モードの切り替え。カウント停止やテストモードを含む。|
|+$23|テストレジスタ|LSIの健全性確認用。通常運用では触れてはならない。|
|+$25|RESET コントローラ|分周回路やカウンタの初期化を司る。|

これらのレジスタを操作する際、開発者は単なる数値の書き込みではなく、RTC内部の状態遷移（Busyフラグ等）を常に意識したプログラミングを強いられる。

## 4. RTCアクセスプロトコル：時刻データの読み出しと書き込み技術

RTCはCPUと比較して極めて低速なデバイスであり、かつ内部で非同期にカウントアップが行われる。そのため、整合性を保つための「儀式」が必要となる。

### 時刻の読み出し：ティアリング（不整合）の回避

読み出し中に秒が繰り上がることで、例えば「1分59秒」が「1分00秒」や「2分59秒」として誤読されるリスクがある。X68000における標準的な回避策は以下の通りである。

1. **二度読み比較法:** 時刻データを2回連続で読み出し、両者が完全に一致するまでループさせる。
2. **Busyフラグ監視:** RTCのステータスレジスタに存在するBusyビット（更新中フラグ）を確認し、更新が行われていない隙を突いてアクセスする。

### 時計データの書き込み：保護と順序

時刻設定時には、まずMODEレジスタを操作して「書き込み許可状態」へ移行させる必要がある。設定順序は「秒」から上位桁へと進めるのが通例であり、最後にカウント動作を再開させることで、システム全体の時刻整合性を確定させる。

## 5. 実装分析：時計読み出しサンプルプログラムの構造解剖

『Source Context』157ページに示されるサンプルプログラムの構造を、スペシャリストの視点で分析する。

### 制御ポリシーの評価

本書のサンプルプログラムが「割り込みを使用せず、ステータスチェック（ポーリング）で動作を待つ」方針を採っているのは、極めて理にかなっている（6ページ）。RTCのような低速デバイスでは、割り込みハンドラのオーバーヘッドやスタック消費、ベクタ競合の不確実性を排除し、確実なデータ取得を優先すべきだからである。

### C言語（gcc/XC）によるセーフアクセス

I/Oレジスタへのアクセスにおいて、`volatile` 修飾子の欠如はコンパイラの最適化による「アクセス消失」を招く。特に、時刻が更新されるのを待つループにおいて、レジスタが一度しか読み出されないといった致命的なバグを防がねばならない。

```c
/* X68000 RTC アクセスの極意 */
#define RTC_BASE 0xE8A001
typedef volatile unsigned char * ioreg;

void read_rtc_seconds() {
    ioreg sec_units = (ioreg)(RTC_BASE + 0x00);
    unsigned char s1, s2;

    /* ティアリング回避のための二度読み */
    do {
        s1 = *sec_units & 0x0F;
        s2 = *sec_units & 0x0F;
    } while (s1 != s2);
    
    /* 取得したs1を処理に回す */
}
```

### 結び

RTC制御を完全に掌握することは、X68000のハードウェア・プログラミングにおける登竜門である。ALARM信号によるMFPへの介入、奇数アドレスへの精密なアクセス、そして非同期更新を克服する読み出しロジック。これら一つひとつの積み重ねが、堅牢なシステムソフトウェアの基盤となる。読者諸氏には、この基礎を足掛かりとして、さらに複雑な画面制御やDMA転送の深淵へと進まれることを切に願う。
