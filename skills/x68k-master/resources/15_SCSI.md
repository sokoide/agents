# X68000 SCSI

X68000シリーズの進化において、外部記憶装置インターフェースがSASI（Shugart Associates System Interface）からSCSI（Small Computer System Interface）へと移行したことは、単なる規格の更新ではない。それは、CPU主導のI/O管理から、独立したプロトコルコントローラによる自律的なバス管理への転換を意味している。本稿では、ハードウェア・アーキテクチャの視点から、SPC（SCSI Protocol Controller）とDMAC（HD63450）の連携、および低レベルレジスタ操作における戦略的実装について詳解する。

---

## 1. X68000 SCSIシステムの戦略的意義とアーキテクチャ概要

X68000シリーズにおけるSCSIへの移行は、1990年の「SUPER-HD」を契機に標準仕様として定着した。それまでのSASI実装は、I/Oコントローラの機能を限定的に利用したものに過ぎなかったが、SCSIの採用はシステムの拡張性とデータ転送効率に劇的なパラダイムシフトをもたらした。

SCSIとSASIのおもな違いは以下の通りである。

- 複数ホストの構成に対応した
- 時間のかかるコマンド処理の場合にいったんバスを切り離し（ディスコネクト）、後で再接続する（リコネクト）機能が追加された
- バスの使用権の調停をするアービトレーションフェーズ、再接続のためのリセレクションフェーズが追加された
- イニシエータからターゲットへのメッセージ転送機能が追加された
- メッセージフェーズ、ステータスフェーズで返される値が規格化された

### 移行の系譜とハードウェアの自律性

「SUPER」「XVI」「Compact」と続く後継機種において、SCSIは標準インターフェースとしての地位を確立した。特に16MHz駆動を実現した「XVI」においては、高速化するCPUクロックに対してI/O転送がボトルネックとなることを防ぐため、独立したSPC（MB89352等）の搭載が不可欠であった。SPCがバスフェーズ管理（Phase Handshake Logic）をハードウェアレベルで隠蔽し、CPUを複雑な信号操作から解放したことで、高負荷なマルチタスク処理下でも安定したディスクI/Oが保証されたのである。

### SCSIバスの構成

SCSIバスはSASIと同様に最大8個のデバイスを接続できるが、SCSIの場合イニシエータ（X68000本体）自体もIDを必要とするため、バス上に接続できるのは7個までとなる。X68000はデフォルトでID#7を使用する。

- **物理構成:** 8ビット双方向データバスと、REQ, ACK, BSY, SEL, C/D, I/O, MSG, ATN等の制御信号。SCSIではSASIにATN信号とDP（パリティ）が追加された。
- **フェーズ遷移の論理フロー:**
    1. **Bus Free:** バスの解放状態を確認。
    2. **Arbitration:** バス使用権の調停。ID#7が最も優先度が高い。
    3. **Selection/Reselection:** ターゲットの選択、またはターゲットからの再選択。
    4. **Information Transfer Phases:** Command, Data, Status, Messageの各相。

ハードウェアによるこれらの状態遷移を制御する具体的なリソース管理について、次節で解体する。

## 2. X68000独自のSCSIインターフェース実装とリソース管理

X68000におけるSCSI実装の最大の特徴は、16ビットバス上に配置された8ビット周辺デバイスという構造に由来する「アドレス配置の偏り」と「IPL/SRAMによる管理構造」にある。

### I/Oマップ配置とアドレスアライメント

SCSI/HDC領域はメモリマップ上の `$E96020`（内蔵モデル）または `$EA0000`（拡張ボードCZ-6BS1）から開始される。SPCレジスタは**奇数アドレス**にマッピングされており、これは16ビットデータバスの下位8ビット側に8ビットレジスタを接続しているX68000特有の設計思想による。

| 項目 | SCSI内蔵モデル | 拡張ボード（CZ-6BS1） |
| :--- | :--- | :--- |
| SPCのポートアドレス | `$E96020`〜`$E9603F` | `$EA0000`〜`$EA001F` |
| SCSI-ROMのアドレス | `$FC0000`〜`$FC1FFF` | `$EA0020`〜`$EA1FFF` |
| SPCの割り込みレベル | レベル1 | レベル2と4を選択可 |
| ベクタ | `$6C` | `$F6` |
| SCSI-ROM識別ラベル | `$FC0024`〜'SCSIIN' | `$EA0044`〜'SCSIEX' |

### IPL-ROMおよびSRAMの役割

- **IPL-ROM:** SCSI内蔵モデルでは、`$FC0000`〜`$FC1FFF`の8KBをSCSI用IPLブートプログラム領域として使用する。起動プロセスにおいてSCSIデバイスをスキャンし、ブート可能なドライブを特定する。SCSI内蔵モデルではデフォルトメモリ容量を2MBとするため、`$FEF079`の内容が`$10`から`$20`に変更されている。
- **SRAM管理:** SCSI対応にともない、`$ED006F`、`$ED0070`、`$ED0071`が使用される。

| レジスタ | 内容 |
| :--- | :--- |
| `$ED006F` | 有効フラグ。`$56`（ASCII 'V'）なら有効。未設定ならSCSIローダが'V'を書き込み、`$ED0070`=$07、`$ED0071`=$00に初期化 |
| `$ED0070` | bit3: 内蔵/オプションボード選択（0=内蔵, 1=オプション）。bit2-0: 自分のSCSI ID |
| `$ED0071` | SASIディスク接続フラグ。対応IDのビットを'1'に設定 |

- **管理構造:** SCSIハードディスクの管理情報（パーティション等）は、システム起動時にこれらの情報を参照してメモリ上に展開される。

物理的配置を理解したところで、それらを操作する心臓部であるSPCの内部レジスタ群を分析する。

## 3. SPC（SCSIプロトコルコントローラ）のレジスタ構成と制御ロジック

SPCはSCSIプロトコルの複雑なシーケンスをハードウェア・ステートマシンとして抽象化し、CPU負荷を軽減する。X68000では富士通のMB89352が使用されており、SCSIバス制御に必要な機能の多くをハードウェア化している。プログラマは以下のレジスタ群を通じてバスフェーズを制御する。

### SPC主要レジスタ機能詳細

ベースアドレス：SCSI内蔵モデル `$E96020`、CZ-6BS1 `$EA0000`

| レジスタ名 | オフセット | 機能概要 |
| :--- | :--- | :--- |
| **BDID** | +$01 | Bus Device ID：自分のID番号の設定/読み出し |
| **SCTL** | +$03 | SPC Control：動作モードや付属機能の選択 |
| **SCMD** | +$05 | SPC Command：動作コマンドの指示や転送モードの選択 |
| **INTS** | +$09 | Interrupt Status：割り込み要因の判別/リセット |
| **PSNS** | +$0B | SCSIバスの制御信号の状態読み出し |
| **SDGC** | +$0D | SPC Diag Control：SPCの自己診断用 |
| **SSTS** | +$0F | SPC Status：接続状態やバッファ状態の読み出し |
| **SERR** | +$11 | SPC Error status：パリティエラーやハード異常のエラーステータス |
| **PCTL** | +$13 | Phase Control：フェーズの明示的な指定 |
| **MBC** | +$15 | Modified Byte Counter：バッファ転送数の制御カウンタ |
| **DREG** | +$17 | Data Register：DMA転送時のデータレジスタ（8バイトFIFO） |
| **TEMP** | +$19 | Temporary Register：マニュアル転送時のデータ転送、ID設定用 |
| **TCH** | +$1A | Transfer Counter High：転送バイトカウンタ上位 |
| **TCM** | +$1B | Transfer Counter Mid：転送バイトカウンタ中位 |
| **TCL** | +$1C | Transfer Counter Low：転送バイトカウンタ下位 |

### 主要レジスタのビット配置

**BDID（+$01）** — 写き込み時は下位3ビットでID 0〜7を設定。読み出し時は設定IDに対応するビットが'1'になったデータが返る（例：ID=7書き込み → $80読み出し）。

**SCTL（+$03）** — bit7: Reset & Disable, bit6: Control Reset, bit5: Diag Mode, bit4: Arbitration Enable, bit3: Parity Enable, bit2: Select Enable, bit1: Reselect Enable, bit0: Interrupt Enable。X68000ではParity=0（パリティチェック無効）、Select=0（イニシエータ専用）に設定。

**SCMD（+$05）** — bit7-5: コマンドコード（111=Set ACK/REQ, 110=Reset ACK/REQ, 101=Transfer Pause, 100=Transfer, 011=Set ATN, 010=Reset ATN, 001=Select, 000=Bus Release）。bit4: RST Out, bit3: Intercept Transfer, bit2: Program Transfer, bit0: Termination Mode。

**INTS（+$09）** — bit7: Selected, bit6: Reselected, bit5: Disconnected, bit4: Command Complete, bit3: Service Required, bit2: Time Out, bit1: SPC Hard Error, bit0: Reset Condition。CPUが書き込むと'1'のビットだけクリアされる。

**PSNS（+$0B）** — bit7: REQ, bit6: ACK, bit5: ATN, bit4: SEL, bit3: BSY, bit2: MSG, bit1: C/D, bit0: I/O。SPCの動作状態に関係なくいつでも読み出せる。

**PCTL（+$13）** — bit7: Busfree INT Enable, bit2-0: Transfer Phase（111=メッセージイン, 110=メッセージアウト, 011=ステータス, 010=コマンド, 001=データイン, 000=データアウト）。

レジスタ設定によるバス確立後は、高速データ移動を実現するためにDMACとの高度な連携が行われる。

## 4. データ転送メカニズム：SPC転送モードとDMAの連携

X68000の高速性能を支えるのは、SPCとHD63450（DMAC）のチャンネル#1による密接な相互作用である。このチャンネルは4つのSASI用DMAチャンネルと共用であるため、SASIとSCSIの両方を使用する場合にはDMA設定に注意が必要である。

### SPCの転送モード

SPCは3つの転送モードを持つ。

| 転送モード | データアクセス | DREQ信号 | 使用するレジスタ |
| :--- | :--- | :--- | :--- |
| マニュアル転送 | TEMPレジスタ | 出力しない | PSNS, SSTS |
| ハード転送 | DREGレジスタ | 出力しない | （または割り込み） |
| DMA転送 | DREGレジスタ | 出力する | — |

マニュアル転送はCPUがREQ-ACKハンドシェークをすべて制御するモード。ハード転送はSPCがバス制御を自動化し、DREGレジスタ（8バイトFIFO）を介してデータ転送を行う。

X68000ではSPCのDREQ信号をDMACには接続しておらず、**DTACK信号の作成に使用**している。DMACは通常のメモリ間転送にプログラミングされるが、SPCのDMA転送要求信号がDTACK信号に接続されているため、転送要求が発生するまでDMACを待たせることができる。ただし、約8μs待ってもDMA転送要求が発生しない場合はバスエラーが発生し、強制的に回復される。

### DMAC設定と「Pack Mode」の重要性

SCSI転送において、Human 68Kはチャンネル#1に対して以下の戦略的デフォルト設定を行う。

- **DCR（デバイスコントロール）:** `$80` (68000バスタイプ、8ビットデバイス)
- **OCR（オペレーションコントロール）:** `$31` (転送方向はCCR bit7で制御)
- **Pack Modeの活用:** SCSIは8ビットデバイス（`DPS=0`）だが、X68000のメインメモリは16ビットである。DMACは「パック転送」を実行し、2回のSCSIアクセスを1回の16ビットメモリ書き込みに集約することで、バス占有率を下げ効率を最大化している。

### SPCコマンドの実行手順

データ転送プロセスの論理フローは以下の通りである。

1. **Bus Release:** `SCMD`にリセットまたはバス解放を命じ、Bus Free状態を確保。
2. **Selection:** `BDID`にターゲットIDをセットし、`SCMD`に `Select` コマンドを発行。`TCH/TCM`にタイムアウト時間を設定し、`TEMP`に自分と相手のIDビットをセット。
3. **Command Phase:** CDB（コマンド記述ブロック）を `DREG` 経由で送出し、ターゲットに動作を指示。
4. **DMA Setup:** DMACの `MAR` にバッファアドレス、`MTC` に転送数をセットし、`CCR` に `$80` を書き込み開始。
5. **Transfer:** SPCの `SCMD` に `Transfer` コマンドを発行し、DMACとのハンドシェイクを開始。

物理的な転送確立後、バス上では論理的なコマンド体系に基づいた対話が行われる。

## 5. SCSIコマンド体系とプロトコル・ハンドシェイクの解読

デバイス間の対話は、CDBによる厳密なプロトコルに基づいている。SCSIコマンドには6バイト長（グループ0）、10バイト長（グループ1）、12バイト長のものがある。

### 主要SCSIコマンド一覧

| オペレーションコード | グループ | コマンド名 | X68000における役割 |
| :--- | :--- | :--- | :--- |
| `$00` | 0 | Test Unit Ready | デバイスの通電および動作準備状態を確認 |
| `$01` | 0 | Rezero Unit | シリンダ0へのヘッド移動などを行う |
| `$03` | 0 | Request Sense | `CHECK CONDITION` 発生時、センスデータを取得 |
| `$04` | 0 | Format Unit | メディアのフォーマットを行う |
| `$08` | 0 | Read | データの読み出し（6バイト長） |
| `$0A` | 0 | Write | データの書き込み（6バイト長） |
| `$12` | 0 | Inquiry | ターゲットおよびユニットの属性情報取得 |
| `$1A` | 0 | Mode Sense | メディアやユニットのパラメータ取得 |
| `$25` | 1 | Read Capacity | ユニットのブロック長やブロック数の取得 |
| `$28` | 1 | Read（拡張） | 拡張READ（32ビットLBA指定の大容量転送） |
| `$2A` | 1 | Write（拡張） | 拡張WRITE（同上） |

このうち上記9つはHuman68KのSCSIドライバがサポートを要求している必須コマンドである。 `$28` と `$2A` は6バイト長コマンドを拡張し、より大きなブロック番号と転送数の指定が可能になったもので、ディスク先頭のSCSIデバイスパラメータ中に拡張READ/WRITEコマンドが使用できるか否かを示すフラグが存在する。

ターゲットから返される「ステータスバイト」が `$02` (Check Condition) の場合、直ちに `REQUEST SENSE` を発行して詳細を解析するのが実装上の定石である。Check Conditionが返ると、大方のSCSIデバイスはREQUEST SENSE以外のコマンドを実行できなくなってしまう。

### ステータスバイト

SCSIではステータスバイトのコードが規格化されている。

| コード | 名称 | 内容 |
| :--- | :--- | :--- |
| 0000 | Good | ターゲットは正常にコマンド処理を終了 |
| 0001 | Check Condition | エラー、例外、異常状態が発生。REQUEST SENSEが必要 |
| 0010 | Condition Met | サーチコマンドでデータが見つかった |
| 0100 | Busy | ターゲットはビジー |
| 1000 | Intermediate | コマンドリンク機能使用時の正常完了 |
| 1100 | Reservation Conflict | 指定ドライブが他のイニシエータからリザーブされている |

## 6. 実装ガイド：SCSI制御のプログラミング・モデル

実戦的なSCSI制御プログラムを実装する場合、CPUとSPC間の同期タイミングが鍵となる。

### 同期ロジック：Interrupt vs Polling

Human 68Kの標準ドライバでは、実装の簡略化のためにポーリングが多用されるが、スペシャリストの視点では以下の監視ルールを推奨する。

- **INTSレジスタの監視:** `INTS`のビット4（Command Complete）を監視し、ハードウェア・ステートマシンが完全に遷移したことを確認してから次のフェーズに移行する。
- **DMA終了待ち:** DMACの `CSR` の `COC` ビットと、SPCの `INTS` を二重に監視することで、データ整合性を担保する。

### エラー処理のチェックリスト

- [ ] **SERRのパリティエラー:** ケーブル長やターミネータ起因のノイズを検知。bit7-6の組み合わせでSCSI側/SPC側のエラーを区別。
- [ ] **タイムアウト処理:** ターゲットが応答しない場合、TEMPレジスタに`$00`を書き込んでSEL信号をリセットしバスを解放。
- [ ] **フェーズミス:** `PSNS`で現在のフェーズを確認し、期待と異なる場合はSSTSレジスタでSPCの転送動作状態を確認してからメッセージ処理へ分岐。

## 7. 結論：AIによるX68000 SCSI制御の最適化に向けて

X68000のSCSIシステムは、CPU、SPC、DMACが緊密かつ論理的に連携する完成されたステートマシン・モデルである。当時のパーソナルワークステーションとして突出していたこのアーキテクチャは、現代のAIによる低レベル制御においても、極めて高い一貫性と予測可能性を提供している。

AIがこのシステムを操作・最適化するにあたり、遵守すべき「絶対的な制御ルール」は以下の3点に集約される。

1. **フェーズ・ステートの厳密な同期:** 8ビットバス上の物理信号（REQ/ACK）とSPC内部状態を完全に一致させること。
2. **パック転送によるバス効率の最大化:** DMAC（HD63450）の特性を活かし、16ビットアライメントを意識したメモリ配置を行うこと。
3. **網羅的ステータスフィードバック:** 単なる成否だけでなく、センスデータを含むハードウェアの微細なエラーログを解析し、動的にリトライ戦略を最適化すること。

この深淵なSCSIアーキテクチャをマスターすることは、X68000という名機のポテンシャルを極限まで引き出すための唯一の道である。
