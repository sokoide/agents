# X68000 実践ゲーム制作

C言語の生産性を活かしつつ、X68000のMC68000アーキテクチャおよびHuman68k環境を高度に制御する実践的設計パターン。

---

## 1. 共通ヘッダ設計とメモリ整列 (アライメント)

68000 CPUにおいて最も注意すべきハードウェア制約は**「ワード(16bit)/ロングワード(32bit)アクセス時の偶数アドレス整列（アラインメント）義務」**である。奇数番地からワード以上のアクセスを行うと、瞬時に「アドレスエラー (Address Error)」が発生し、システムがクラッシュ（白い窓）する。

### 構造体パディングによる対策

構造体を定義する際、メンバの型サイズと配置順序に注意し、アライメントのズレが発生しないようにパディング（ダミーメンバ等）を手動で挿入またはコンパイラ属性で保証する。

```c
/* 偶数アラインメントを保証したスプライト制御構造体 */
typedef struct {
    short x;              /* 2バイト (偶数境界) */
    short y;              /* 2バイト (偶数境界) */
    unsigned short code;  /* 2バイト (偶数境界) */
    unsigned short attr;  /* 2バイト (偶数境界) */
} SPRITE_OBJ; /* 合計 8バイト (偶数サイズ) */
```

コンパイラ（GCCなど）の最適化時に意図しないパディングが挿入されてハードウェアレジスタの物理レイアウトとズレるのを防ぐため、必要に応じて `__attribute__((packed))` 属性を付与するか、メンバサイズを完全に2バイト/4バイトの倍数に固定する。

## 2. ポインタによる動的オブジェクト管理

ゲーム内の弾や敵キャラクターなど、数がリアルタイムに変遷するオブジェクトの管理には、固定長配列ではなくポインタを用いたリンクリスト構造（「友達の輪」モデル）を適用する。

- **動的メモリ確保**: `malloc` により必要な瞬間にオブジェクト用メモリを動的に確保。
- **オブジェクト消去**: リンクリストのポインタを付け替えるだけで、高速かつメモリ効率の高いオブジェクト消去が可能。
- 実装コードの具体例は [point.c](file:///Users/scott/repo/sokoide/agents/skills/x68k-master/resources/samples/point.c) を参照。

```c
typedef struct friend_loop {
  char *name;                /* オブジェクト識別子 */
  char *address;             /* 各種属性データ */
  struct friend_loop *next;  /* 次のオブジェクトへのポインタ */
} FRIEND;

/* 新規オブジェクトの追加 */
void add_friend(FRIEND **root, char *name) {
    FRIEND *new_p = (FRIEND *)malloc(sizeof(FRIEND));
    new_p->name = name;
    new_p->next = *root;
    *root = new_p;
}
```

## 3. 当たり判定と処理タイミング設計 (「洗濯機の奥さん」理論)

ゲーム内のオブジェクト衝突判定は、高速な矩形（バウンディングボックス）判定が基本となる。しかし、これらの衝突判定や描画データの作成ロジックを行う「タイミング」が最も重要である。

- **表示期間中 (CPU演算処理)**:
  画面の表示期間中は、CPUはジョイスティック等の入力解析、敵アルゴリズム演算、当たり判定などの「ロジック（重い処理）」を行う。
- **垂直帰線期間 (VBLANK中) (VRAM一括反映)**:
  画面が描画されていない垂直帰線（VBLANK）期間の短い瞬間に、ロジック処理で決定した座標やパターンを一気にスプライトRAMやVRAMに反映（転送）させる。
  
これにより、描画競合による画面のノイズやティアリングを防ぐ。

## 4. 例外処理とエラーハンドリング

特権モードであるスーパーバイザモード（`B_SUPER(0)` または `_iocs_super`）での直接ハードウェア制御は、Human68kによるシステム保護の対象外となる。そのため、デバッグ時や実行中に発生した「バスエラー」や「アドレスエラー」を独自にトラップ（フック）し、エラーが起きたPC（プログラムカウンタ）などのデバッグ情報をログ出力して安全にHuman68kへ復帰（アボート）する仕組みが不可欠となる。

### TRAP 14 (例外ベクタ) のフック

X68000では、バスエラーやアドレスエラーなどの例外処理に `trap #14`（ベクタ `$2e` 相当）が関連している。これを自前のトラップハンドラにフックすることで、システムをフリーズさせることなくクラッシュの原因（PCアドレスなど）を特定できる。

- 実装の全体構造は [trap14.c](file:///Users/scott/repo/sokoide/agents/skills/x68k-master/resources/samples/trap14.c) を参照。
- このエラーハンドラを利用したテストプログラムは [bustrap.c](file:///Users/scott/repo/sokoide/agents/skills/x68k-master/resources/samples/bustrap.c) を参照。

#### trap 14 フックの初期化コード例

```c
#include <interrupt.h>

static void (*trap_14)(); /* 元のトラップベクタ保存用ポインタ */

static void trap14_handler(void) {
    PRAMREG(code, d7); /* 例外コードをレジスタ D7 から取得 */
    PRAMREG(add,  a6); /* エラー発生アドレスを A6 から取得 */
    SET_FRAME(a5);     /* スタックフレーム */
    
    /* エラー内容を trapbuf 等に保存して、安全に exit へジャンプする */
    // ... デバッグログ出力とシステム復旧処理 ...
}

void init_trap14(void) {
    /* 現在の TRAP 14 ベクタ (0x2e) を取得・保存 */
    trap_14 = (void *)INTVCG(0x2e);
    
    /* 自前のハンドラを登録 */
    INTVCS(0x2e, trap14_handler);
}
```

このような例外処理フックを用いることで、X68000の直接ハードウェア操作開発におけるデバッグ効率が飛躍的に向上する。
