# X68000 Master

`references` を正として、X68000 実機向けコードの設計・実装・レビューを行う。
特に 68000 のアドレス整列、Human68k/IOCS、VRAM、DMAC、割り込み、周辺 I/O を安全側で扱う。

## 最初に確認すること

- 対象機種: 初代 / ACE / EXPERT / PRO / SUPER / XVI / Compact / 030
- 開発形態: Human68k ネイティブ gcc/XC か、クロス開発か
- 直接叩く対象: GVRAM / TVRAM / CRTC / DMAC / MFP / 音源 / ストレージ / 通信
- IOCS/DOS 呼び出しで足りるのか、性能要件のために直叩きが必要か

## 出力方針

- 低レベル説明では、必ずどの `references` 以下のファイルに根拠があるか明示する。
- 直接 I/O アクセス例では、Supervisor モード移行と復帰を必ず明示する。
- VRAM と I/O 領域を混同しない。`$E80000-$EBFFFF` はシステム I/O、`$C00000-$E7FFFF` は VRAM 系。
- レジスタ名・アドレス・ビットは、`references` で確認できた範囲だけを書く。怪しい細部は断定しない。
- `references` に明記がない細部は、推測せず「要原典確認」と明示する。

## 共通制約（全デバイス共通）

以下は全デバイスで共通。各リソースファイルでは原則として再説明しない。

1. **Supervisor モード必須**: システム I/O（`$E80000`以降）へのアクセスは `_iocs_super(0)` で Supervisor に移行してから行う。終了時に必ず復帰。
2. **メモリマップト I/O**: 68000 には独立 I/O 空間がない。全ペリフェラルはメモリ空間上のアドレスにマッピングされる。
3. **奇数番地 / バイトアクセス**: 8 ビットペリフェラル（MFP, RTC, SCC, FDC, SASI, SCSI 等）はデータバス D0-D7 に接続されているため、**奇数番地へのバイトアクセス（`move.b` 等）**で操作する。ワードアクセスは上位バイトが不定。
4. **`volatile` 必須**: レジスタポインタには必ず `volatile` を付与。コンパイラ最適化によるアクセス消失を防ぐ。
5. **68000 アラインメント**: ワード/ロングアクセスは偶数アドレス前提。奇数アドレスへのワードアクセスは Address Error。
6. **VRAM と I/O の区別**: `$C00000-$E7FFFF` は VRAM 系、`$E80000-$EBFFFF` はシステム I/O。混同しない。

## システム初期化・終了ライフサイクル

ゲーム等のアプリケーションは、起動時に Human68k の標準機能（カーソル点滅、マウスカーソル、ファンクションキー表示、ソフトキーボード、電卓等）を抑制し、終了時に全て復元するライフサイクルを徹底すること。

1. **初期化順序**: Supervisor移行 → G_CLR_ON → CRTMOD → 画面初期化 → 白帯処理(COPY/NMI) → スプライト初期化
2. **終了・復元順序**: 割り込みベクタ復元 → 画面モード復元 → カーソル/マウス/FK復元 → KFLUSH
3. **COPYキー (trap #12)** と **NMI (trap #31)** のハンドラを登録した場合は、終了時に必ず元のベクタを復元する。
4. VSYNC周波数は高解像度モードで **55.46Hz** (約10%遅い)。60Hz前提のタイミング設計は修正すること。

詳細は `21_システム初期化と終了.md` を参照。

### V-Sync 同期手法の使い分け

スプライト・スクロールの実装では、以下 3 種の V-Sync 同期手法をサンプルで示している。用途に応じて推奨する手法を選択する。

| 手法 | サンプル | 推奨用途 |
| --- | --- | --- |
| VDISPST 割り込み | `sp_test0.c`, `map.c` | 最も安全で推奨。表示期間中のハードウェアレジスタ更新を確実に同期 |
| ONTIME タイマー | `sp_test1.c` | 割り込みハンドラの登録を避けたい場合。精度はやや劣る |
| MFP ポーリング | `sp_test.c`, `map_b.c` | IOCS に依存しない最下層の手法。学習・デバッグ用途 |

## 実装ルール

1. GVRAM / TVRAM はメモリ空間であり、I/O 領域とは別物として扱う。
2. 大量転送は CPU ループより DMAC を優先する。特に FDD/HDD/ADPCM は DMAC 前提。
3. 割り込みベクタや MFP 設定を横取りしたら、終了時に必ず復元する。
4. パレット・表示タイミング・ラスタ制御は CRTC/ビデオ制御とセットで扱う。
5. キーボードは MFP 内蔴 USART 系。SCC は主に RS-232C/マウス側。
6. ジョイスティックは i8255 / I/O コントローラ系。単純なメモリ読み出し 1 本で決め打ちしない。
7. 怪しい場合は IOCS/DOS 優先。実機依存の直叩きは根拠があるときだけ出す。
8. スプライトを描画する際は、走査線との干渉を避けるため仮想SAT（メモリバッファ）でのダブルバッファリングとV-Sync同期（VDISPST割り込み、ONTIMEタイマー、またはMFPポーリング）を徹底する。
9. 直接ハードウェア制御を行う場合は、バスエラーやアドレスエラーをデバッグしやすくするため、TRAP 14 (例外ベクタ 0x2e) をフックして独自の例外ハンドラを構築する。
10. **elf2x68k の IOCS ライブラリ関数は正しく実装されている**（逆アセンブル検証済み）。動かない場合は IOCS 呼び出し以外の原因（PCG パッキング方式等）を疑うこと。raw `trap #15` は代替手法として利用可能。
11. **G_CLR_ON (IOCS 0x90) を忘れるとGVRAM描画が表示されない**。グラフィック画面を使用する場合は必ず呼び出す。
12. **16/256/65536色モードどの場合もGVRAMは1ドット=2バイト**。65536色モード以外では上位のビットが無視される。
13. **入力処理**: キーボードは BITSNS ($04) でグループ単位に読む。ジョイスティックは JOYGET ($3B) で取得後、NOTで正論理化。ボタンの「押した瞬間」はエッジ検出（XOR+AND）で抽出。詳細は `20_入力処理.md`。
14. **ゲームループ**: 入力→論理→描画準備→VSYNC待ち→ハードウェア反映の順。表示期間中にハードウェアレジスタを書き換えない。詳細は `19_ゲームアーキテクチャ.md`。
15. **スプライト管理**: 仮想SAT (disp_sp[128]) でダブルバッファリング。VSYNC割り込み内でハードウェアSAT ($EB0000) に一括転送。PCG定義もVSYNC内で行う。詳細は `19_ゲームアーキテクチャ.md`。
16. **当たり判定**: 矩形判定 (`|dx| < threshold && |dy| < threshold`)。イテレーション中のスプライト削除は遅延させ、VSYNC経過後に一括実行。
17. **システム初期化・終了**: 起動時にHuman68k標準機能(カーソル、マウス、FK等)を抑制。COPYキー(trap #12)とNMI(trap #31)の無害化。終了時にすべて復元。詳細は `21_システム初期化と終了.md`。
18. **VSYNC周波数は55.46Hz** (高解像度モード)。60Hzより約10%遅い。タイミング設計で考慮すること。

## レビュー観点

- Supervisor モードが必要な I/O 領域を User モードで触っていないか
- VRAM とシステム I/O の境界を取り違えていないか
- 16bit/32bit アクセスの整列違反がないか
- DMAC のチャネル割り当てとチャネルベースを誤っていないか
- MFP/割り込みベクタを変更して復元漏れしていないか
- キーボード、SCC、RTC、i8255 を混同していないか
- 多数のスプライトを動的に更新する際、表示期間中に直接SATを書き換えてティアリング（表示崩れ）を起こしていないか
- スプライト座標の球面スクロール（マスク演算）の考慮漏れがないか
- 例外ハンドラが登録された場合、プログラム終了時に元のトラップベクタが復元されているか
- elf2x68k 環境で IOCS 呼び出しが動かない場合、ライブラリ関数ではなく PCG パッキング等の別原因を調査しているか
- グラフィック画面を表示する際に G_CLR_ON (0x90) を呼び出しているか
- GPALET の IOCS 番号が 0x94（正）であるか。0x11 は誤り
- 16/256/65536色のどのモードでも GVRAM ストライドを 1024バイトとしているか
- 初期化で抑制した機能（カーソル、マウス、FK等）が終了時にすべて復元されているか
- COPYキー (trap #12) と NMI (trap #31) のハンドラが終了時に元のベクタに復元されているか
- ジョイスティック入力でエッジ検出（押した瞬間/離した瞬間）を正しく実装しているか
- VSYNC周波数が55.46Hz（高解像度）であることを考慮したタイミング設計か
- スプライトのVSYNC内一括転送（ダブルバッファリング）を徹底しているか
- 当たり判定ループ内でのスプライト即時削除（リスト破壊）をしていないか
